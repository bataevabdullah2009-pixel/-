# Голосовой журнал продаж

## Назначение

Сохранять проект сфокусированным на проверяемом пути:

```text
Голос Telegram → webhook или polling → STT → LLM-парсер → позиции → Supabase → отчёт
```

Продукт является журналом продаж, а не CRM, складом, кассой или платёжной системой.

## Обязательное чтение

Перед изменением кода прочитать по порядку:

1. `../../../README.md`;
2. `../../../docs/overview/README.md`;
3. `../../../docs/specs/global.md`;
4. `../../../docs/specs/technical/README.md`;
5. `../../../docs/rules/README.md`;
6. каждый `../../../docs/plans/active/*/plan.md`;
7. этот файл.

## Рабочие процессы

### Парсер

Прочитать спецификацию LLM-парсера, реализацию и регрессионные тесты. Сохранять строгий JSON/Zod, проверку по расшифровке, порядок всех позиций, запрет выдуманных значений, порог уверенности и расчёт итогов приложением.

### Webhook

Прочитать спецификации webhook и развёртывания. Сохранять `POST /api/telegram/webhook`, Node.js, безопасное сравнение секрета и общий `processTelegramUpdate`. Не запускать polling на Vercel и не печатать секреты.

### Интерфейс

Прочитать спецификации мобильного интерфейса, API и отчёта. Сохранять серверные компоненты для чтения, серверные действия для мутаций, мобильные карточки, таблицу на компьютере, доступные формы и отсутствие горизонтального переполнения.

### База данных

Прочитать схему, модель данных и все миграции. Для каждого изменения схемы создавать новую миграцию. Применять её до зависимого кода, сохранять service role на сервере, RLS, аудит и мягкое удаление.

### Git и GitHub

Прочитать `../../../docs/rules/git-and-github.md`. Не создавать коммит, не отправлять ветку и не открывать Pull Request без прямого запроса. Не включать несвязанные изменения или секреты.

### Документация

Писать проектную документацию на русском языке. Не создавать новые `README.md`; использовать содержательное имя. Не дублировать существующие спецификации и планы. После переименования проверять все ссылки.

## Запреты

- Не добавлять CRM, склад, остатки, кассу, фискализацию или онлайн-оплату.
- Не менять webhook без доказанной причины.
- Не менять парсер без регрессионных тестов.
- Не менять схему без миграции.
- Не коммитить `.env.local` и секреты.
- Не передавать service role в браузер.
- Не удалять физически голос или аудио при исключении позиции.
- Не утверждать, что внешняя проверка выполнена, если она не проводилась.

## Завершение

1. Обновить профильные спецификации, функции, истории и правила.
2. Обновить `CHANGELOG.md` и активный план.
3. При полном завершении оформить результат в `docs/plans/completed`.
4. Выполнить `npm run lint`, `npm run test`, `npm run build`.
5. Для интерфейса выполнить браузерную проверку.
6. Для интеграций записать порядок миграции и развёртывания и непроверенные внешние шаги.
