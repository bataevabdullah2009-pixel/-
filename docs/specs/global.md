# Global Spec

## 1. Название

`voice-sales-log` - голосовой журнал продаж для небольшого магазина.

## 2. Главная идея

Продавец не заполняет CRM, склад или кассу. Он просто отправляет голосовое сообщение в Telegram.

Система превращает голос в текст, извлекает позиции продажи, сохраняет запись и показывает владельцу простой отчёт:

| Товар | Количество | Выручка |
| --- | ---: | ---: |
| Хлеб | 10 | 400 ₽ |
| Молоко | 7 | 630 ₽ |
| Чай | 3 | 450 ₽ |
| Итого | 20 | 1 480 ₽ |

## 3. Проблема

В небольших магазинах продажи часто записываются в тетрадь, личные сообщения или голосовые заметки. Из-за этого:

- данные сложно найти;
- итоги дня считаются вручную;
- спорные записи теряются;
- владелец не видит простую картину за день, неделю или месяц.

## 4. Решение

Проект даёт один понятный поток:

```text
Telegram voice -> STT -> cleaned text -> sale items -> database -> report
```

Владелец видит:

- что продано;
- сколько продано;
- какая выручка;
- что нужно проверить вручную.

## 5. Роли

| Роль | Цель |
| --- | --- |
| Продавец | Быстро зафиксировать продажу голосом. |
| Владелец | Смотреть записи и отчёты без ручного подсчёта. |
| Преподаватель | Проверить архитектуру, запуск, тесты и границы проекта. |

## 6. MVP Scope

В MVP входят:

- Telegram-бот.
- Приём голосовых сообщений.
- STT-распознавание.
- LLM cleanup.
- LLM parser в строгий JSON.
- Supabase Storage для аудио.
- Supabase Postgres для данных.
- Простая таблица `products` с ценой по умолчанию.
- Таблицы `voice_records`, `sales`, `sale_items`.
- Веб-панель записей.
- Веб-панель отчёта.
- Фильтры по датам.
- Фильтр по продавцу.
- Поиск по тексту.
- Блок “Нужно проверить”.
- Ручное исправление спорной позиции.
- Zod-валидация.
- Vitest-тесты.

## 7. Out Of Scope

Не входит:

- склад;
- остатки;
- касса;
- фискализация;
- онлайн-оплата;
- штрихкоды;
- CRM;
- клиентская база;
- сложная аналитика;
- автоматическое придумывание цен;
- полноценный каталог товаров.

## 8. AI Contract

ИИ используется только для:

1. очистки текста;
2. добавления пунктуации;
3. извлечения структурированных позиций из уже сказанного текста.

ИИ запрещено:

- придумывать товар;
- придумывать цену;
- придумывать количество, если смысл не следует из текста;
- добавлять категории;
- считать выручку по неизвестной цене;
- превращать запись в аналитику.

## 9. Parser JSON Contract

```json
{
  "items": [
    {
      "product_name": "Хлеб",
      "quantity": 3,
      "unit": "шт",
      "price": 40,
      "total": 120,
      "confidence": 0.95
    }
  ],
  "raw_text": "хлеб 3 по 40",
  "cleaned_text": "Хлеб - 3 штуки по 40 рублей.",
  "needs_review": false
}
```

Если цена не названа:

```json
{
  "items": [
    {
      "product_name": "Хлеб",
      "quantity": 3,
      "unit": "шт",
      "price": null,
      "total": null,
      "confidence": 0.9
    }
  ],
  "raw_text": "хлеб 3 штуки",
  "cleaned_text": "Хлеб - 3 штуки.",
  "needs_review": true
}
```

## 10. Статусы

### Voice record and sale

- `pending` - запись создана, обработка не завершена.
- `processed` - запись обработана нормально.
- `needs_review` - текст или позиции нужно проверить.
- `failed` - обработка упала.

### Sale item

- `processed` - товар, количество и цена известны.
- `needs_price` - товар распознан, цена неизвестна.
- `needs_review` - низкая уверенность или спорная позиция.
- `failed` - позиция не обработалась.

## 11. Report Contract

Основная таблица отчёта показывает только:

- товар;
- количество;
- выручку.

В основной таблице нельзя показывать:

- имя продавца;
- время каждой продажи;
- технические статусы;
- длинный raw transcript;
- служебные id.

## 12. Acceptance Summary

Проект считается готовым для учебной сдачи, если:

- `npm install` проходит;
- `npm run lint` проходит;
- `npm run test` проходит;
- `npm run build` проходит;
- бот принимает голосовое;
- запись появляется в базе;
- продажа и позиции сохраняются;
- отчёт группирует одинаковые товары;
- позиции без цены попадают в “Нужно проверить”;
- документация организована и заполнена;
- секретов в коде нет.
