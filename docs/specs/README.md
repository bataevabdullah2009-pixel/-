## Короткое резюме

`voice-sales-log` - это не CRM, не склад, не касса и не товароучётная система.

Главный сценарий:

1. Продавец отправляет голосовое сообщение в Telegram.
2. В production Telegram отправляет update на Vercel webhook route; локально polling использует тот же processor.
3. Бот скачивает аудио и сохраняет его в Supabase Storage.
4. STT превращает голос в текст.
5. LLM очищает текст и возвращает строгий JSON с позициями продажи.
6. Система сохраняет голосовую запись, продажу и позиции.
7. Владелец смотрит записи и простой отчёт по товарам, количеству и выручке.

## MVP

В MVP входят:

- Telegram-бот.
- Vercel webhook route для Telegram updates.
- Локальный polling через `npm run bot:dev`.
- Приём voice messages.
- STT-распознавание.
- Очистка текста.
- Извлечение товарных позиций.
- Supabase Postgres и Storage.
- Таблицы `shops`, `sellers`, `products`, `voice_records`, `sales`, `sale_items`, `audit_logs`.
- Веб-панель `/records`, `/sellers`, `/daily-report`.
- Фильтры по датам, продавцу и тексту.
- Группировка одинаковых товаров в отчёте.
- Блок “Нужно проверить”.
- Ручное исправление спорной позиции.
- Zod-валидация и минимальные тесты.

## Не входит в MVP

- Складские остатки.
- Касса.
- Онлайн-оплата.
- Фискализация.
- Штрихкоды.
- CRM.
- Клиентская база.
- Автоматическое придумывание цен.
- Категории и сложная аналитика.

## Обязательные правила

- ИИ не придумывает товары.
- ИИ не придумывает цены.
- Если цена неизвестна, позиция получает `needs_price`.
- Позиции без цены не попадают в выручку.
- Основная таблица отчёта показывает только товар, количество и выручку.
- Все спорные данные доступны для ручной проверки.
