# Интеграционные и браузерные тесты

## Контур

Тесты используют отдельную временную Supabase или локальный stack, фиксированные seed-данные и моки Telegram/STT/LLM. Они не обращаются к production и не требуют production-секретов.

## Обязательные сценарии

- **Webhook:** валидный и неверный secret, неверный JSON, ошибка обработчика, повтор и конкурентный update.
- **Voice processing mock:** загрузка тестового аудио, mock STT, mock LLM, проверка статусов и сохранённых диагностических данных.
- **Report calculation:** данные из тестовой БД, периоды, группировка, неизвестная цена, низкая confidence и мягкое удаление.
- **Manual review:** правка `needs_review`/`needs_price`, пересчёт, аудит, исключение, восстановление и сброс одного дня.
- **Mobile UI:** страницы записей, продавцов и отчёта на ширине 360–390 px, отсутствие горизонтального overflow, формы, состояния loading/empty/error.
- **Isolation:** owner/seller одного магазина не получает данные другого.

Каждый тест очищает собственные данные и не зависит от порядка запуска. Время и внешние ответы фиксируются детерминированно.

## Критерии приёмки

- критический путь webhook → sale → report проходит в CI;
- повтор update не создаёт дубль;
- отказ середины транзакции не оставляет частичных данных;
- ручная правка и аудит проверяются на реальной тестовой схеме;
- мобильный E2E проходит в поддерживаемом браузере без console errors;
- негативные auth/RLS тесты доказательно блокируют чужой `shop_id`;
- flakiness отслеживается, а retry не маскирует системный дефект.
