# Транзакции

## Требование

`sale` и все связанные `sale_items` должны сохраняться атомарно. Голосовая запись, статус, производный итог и обязательный аудит также должны переходить между согласованными состояниями по документированным границам транзакции.

Рекомендуемый контракт — серверная SQL/RPC-функция, которая принимает проверенный результат парсера и в одной транзакции:

1. проверяет ключ идемпотентности;
2. создаёт или обновляет `voice_record`;
3. создаёт `sale`;
4. создаёт все `sale_items`;
5. пересчитывает итог и статусы;
6. пишет обязательный аудит;
7. выполняет commit только после успеха всех шагов.

STT, LLM и загрузка аудио являются внешними вызовами и не удерживают транзакцию БД. Для них нужна явная state machine и очистка/повтор, а не длинная транзакция.

Ручная правка, исключение, восстановление и сброс дня также должны выполнять изменение, пересчёт и аудит атомарно.

## Критерии приёмки

- сбой вставки любой позиции откатывает всю продажу;
- не существует `sale` без ожидаемого набора позиций из успешной операции;
- итог и статусы соответствуют committed-позициям;
- мутация без записи обязательного аудита не фиксируется частично;
- повтор после rollback безопасен благодаря идемпотентности;
- интеграционные тесты искусственно прерывают каждый критический шаг.
