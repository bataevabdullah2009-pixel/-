# Стратегия тестирования

## Автоматические проверки

```bash
npm run lint
npm run test
npm run build
```

- Тесты парсера проверяют признаки чисел, порядок позиций, запрет догадок и низкую уверенность.
- Тесты отчёта проверяют нормализацию, группировку, неизвестную цену, итоги и мягкое удаление.
- Тесты журнала и аудио проверяют маскирование и резервный путь ffmpeg.
- Изолированные тесты route для секрета, неверного JSON и ошибки обработчика отсутствуют.
- Мутации Supabase требуют тестовой БД или моков клиента и сейчас интеграционно не покрыты.
- Автоматические браузерные тесты и CI отсутствуют.
- Нет теста, подтверждающего обязательный фильтр `shop_id` при сбросе дня.
- Нет тестов сетевого исключения LLM, сбоя аудита после сохранения и явного поведения демонстрационного режима.

## Ручной сценарий

1. Отправить голосовое «Сникерс 5 штук по 60 рублей».
2. Проверить ответ, голосовую запись, продажу и позицию.
3. Открыть дневной отчёт; ожидать 5 и 300 ₽.
4. Изменить цену на 70; ожидать 350 ₽ и аудит.
5. Исключить позицию; ожидать 0 ₽ и сохранённую исходную запись.
6. Восстановить; ожидать 350 ₽.
7. Сбросить день; проверить неизменность другой даты.
8. На ширине 390 px проверить отсутствие переполнения, панели ошибки и ошибок консоли.

Модульные тесты не подтверждают рабочие Telegram, Vercel или Supabase.
