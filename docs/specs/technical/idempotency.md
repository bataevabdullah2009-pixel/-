# Идемпотентность Telegram updates

Telegram может доставить один update повторно из-за таймаута, повторной попытки или неоднозначного ответа webhook. Повтор не должен создавать вторую продажу.

## Контракт

- хранить стабильный `update_id` и идентификатор сообщения;
- `telegram_message_id` должен участвовать в уникальном ограничении, достаточном для выбранной модели продавцов/чатов;
- резервировать ключ до запуска дорогих STT/LLM операций;
- конкурентные запросы с одним ключом должны сходиться к одной записи;
- завершённый повтор возвращает безопасный ранее сохранённый результат;
- повтор после ошибки следует явной политике retry, не создавая параллельный бизнес-набор.

Миграция уникальности обязана предварительно проверить существующие дубли и определить способ их устранения.

## Критерии приёмки

- два последовательных одинаковых update создают одну голосовую запись и одну продажу;
- два конкурентных одинаковых запроса дают тот же результат;
- `telegram_message_id` защищён уникальным ограничением в БД;
- повтор не вызывает второй STT/LLM после успешной обработки;
- retry после контролируемой ошибки не создаёт частичные дубли;
- поведение проверено интеграционным тестом webhook и БД.
