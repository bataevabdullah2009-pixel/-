# Авторизация и изоляция магазинов

## Назначение

Авторизация доказывает, кто выполняет запрос, а изоляция определяет, данные какого магазина ему доступны. Наличие `shop_id` в таблице само по себе не является защитой.

## Роли

- **Owner:** проходит серверную проверку Telegram Mini App init data или эквивалентную production-аутентификацию, связан с одним или несколькими магазинами и выполняет разрешённые чтения и мутации.
- **Seller:** определяется по Telegram ID, связан с одним активным `shop_id`, создаёт голосовые продажи, но не получает административный доступ.

## Изоляция по `shop_id`

Сервер выводит `shop_id` только из проверенной личности и серверной связи owner/seller. Клиентский query, form field или JSON не считается источником полномочий. Каждое чтение, update, RPC, подписанная ссылка и аудит ограничиваются тем же магазином.

## RLS

- убрать демонстрационные политики анонимного чтения до production;
- включить deny-by-default для бизнес-таблиц и Storage;
- разрешить владельцу строки только связанных магазинов;
- разрешить продавцу минимальный набор операций, необходимый для приёма сообщения;
- service role оставлять только на сервере и не использовать как замену авторизации;
- проверять положительные и отрицательные сценарии политик.

## Риск текущего режима

Web App не подтверждает владельца, а некоторые операции не имеют обязательного фильтра магазина. Это допускается только с тестовыми данными. Особо опасен сброс дня: до hardening его нельзя использовать в многомагазинном режиме.

## Критерии приёмки

- запрос без валидной сессии отклоняется;
- owner магазина A не читает и не меняет данные магазина B;
- seller не получает owner-права;
- подмена `shop_id` клиентом не меняет область доступа;
- RLS-тесты покрывают все бизнес-таблицы и аудио;
- service role отсутствует в клиентском bundle;
- аудит содержит идентификатор действующего пользователя и `shop_id`.
