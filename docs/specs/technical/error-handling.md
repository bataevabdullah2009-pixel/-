# Обработка ошибок

## Журналирование

Журналируются метаданные Telegram-файла, диагностика преобразования, статус STT, этап парсера, ошибки Supabase и необработанные ошибки бота или webhook. Секреты маскируются. Webhook не возвращает стек клиенту.

HTTP-ответ LLM с ошибочным статусом при очистке текста использует локальную очистку. Сетевое исключение `fetch` сейчас не перехватывается внутри сервиса и завершает весь голосовой конвейер ошибкой.

## Сообщения продавцу

- начало: «Голосовое получено, обрабатываю»;
- успех: очищенный текст и итоговый статус;
- ошибка: короткое безопасное сообщение без технических деталей.

## Выбор статуса

- `failed`: скачивание, STT, разбор или сохранение бросили исключение;
- `needs_review`: низкая уверенность, пустое имя, отсутствующее явное количество или неоднозначность;
- `needs_price`: товар и количество известны, цена не названа и не найдена;
- `processed`: обязательные данные допустимы, итог пересчитан приложением.

## Аудит и частичные ошибки

Обработка и мутации записываются в `audit_logs`. Ошибка аудита после успешной мутации возвращается как частичная ошибка: данные изменены, аудит не записан. Общей транзакции для многотабличных операций сейчас нет.

Чтение веб-панели обрабатывает отсутствие конфигурации как демонстрационный режим, а многие ошибки Supabase — как пустой результат. Пользователь не отличает отсутствие данных от отказа сервиса. До реальной эксплуатации нужен явный режим и видимое состояние ошибки.
