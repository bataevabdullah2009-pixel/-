# Безопасность и окружение

## Файлы и Vercel

- `.env.local` содержит реальные значения и не коммитится;
- `.env.example` содержит только имена и безопасные значения по умолчанию;
- переменные Vercel задаются в настройках проекта;
- после изменения окружения требуется повторное развёртывание.

## Секреты только для сервера

- `SUPABASE_SERVICE_ROLE_KEY` используется сервисами бота, route и серверными действиями;
- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_WEBHOOK_SECRET`, `STT_API_KEY`, `LLM_API_KEY` используются только сервером;
- отсутствующий или неверный секрет webhook возвращает 401;
- журналы проходят рекурсивное маскирование.

## Supabase

RLS включён, но начальные демонстрационные политики разрешают анонимное чтение бизнес-таблиц. До реального использования необходимо:

1. проверять Telegram Web App init data на сервере;
2. определять владельца и магазин;
3. удалить глобальные анонимные политики;
4. ограничить каждый запрос и мутацию магазином владельца;
5. не считать скрытый URL способом авторизации.

Серверные действия сейчас не имеют авторизации владельца и не являются безопасным публичным административным API.

Сброс дня не фильтрует продажи по магазину и при текущем service role может затронуть все магазины выбранного диапазона. Встроенные демонстрационные данные включаются неявно при отсутствии конфигурации Supabase; рабочее окружение должно завершаться явной ошибкой вместо маскировки проблемы.
