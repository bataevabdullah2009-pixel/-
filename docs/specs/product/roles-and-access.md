# Роли и доступ

## Продавец

Продавец использует только Telegram-бота:

- отправляет `/start`;
- отправляет голосовые продажи;
- получает безопасный статус обработки;
- не видит внутренние исключения, stack trace и системные ошибки;
- не получает секреты, ключи или административные ссылки;
- не редактирует базу и отчёты напрямую.

В production Telegram ID продавца должен быть связан с активным продавцом и конкретным `shop_id`. Неизвестный или отключённый продавец не должен создавать продажу.

## Владелец

Владелец открывает Web App и:

- смотрит отчёты по периодам;
- смотрит журнал голосовых записей;
- слушает аудио по короткоживущей подписанной ссылке;
- исправляет позиции `needs_review` и `needs_price`, а также другие активные позиции;
- исключает и восстанавливает позиции;
- видит список продавцов своего магазина.

В production владелец обязан проходить отдельную серверную авторизацию. Каждое чтение и изменение должно выводить разрешённый `shop_id` из проверенной сессии, а не принимать его как доверенный параметр клиента.

## Куратор

Куратор:

- проверяет структуру проекта;
- начинает с корневого `README.md` и `docs/INDEX.md`;
- проверяет спецификации, правила, планы и отсутствие противоречий;
- запускает локальные команды качества;
- проходит описанный демонстрационный сценарий;
- проверяет, что внешние шаги не названы выполненными без доказательств.

Куратору не требуются реальные секреты.

## Система

Система:

- принимает Telegram webhook с секретным заголовком;
- скачивает и сохраняет аудио;
- запускает STT;
- запускает LLM-парсер и детерминированную проверку;
- пишет результат в Supabase;
- строит отчёты из допустимых активных позиций;
- возвращает пользователю безопасный статус.

## Текущий режим

Текущий MVP работает в демонстрационном режиме. Надёжная авторизация владельца и изоляция магазинов обязательны для production. До их реализации разрешены только тестовые данные и контролируемая демонстрация.
