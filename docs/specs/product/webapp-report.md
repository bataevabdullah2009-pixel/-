# WebApp: отчёт по продажам

Статус: реализовано.

## Цель

Дать владельцу магазина мобильный отчёт по распознанным продажам с быстрым контролем выручки, количества и отдельных товарных позиций. Отчёт не заменяет CRM или складской учёт: он показывает фактически сохранённые `sales` и `sale_items` за выбранный период.

## Пользовательские сценарии

1. Владелец открывает Telegram Mini App или разрешённый browser fallback.
2. Выбирает период: сегодня, вчера, неделя, месяц, год или конкретная дата.
3. Видит компактные итоги по активным товарам: выручку и суммарное количество.
4. Просматривает карточки распознанных товаров.
5. Открывает карточку через иконку карандаша, исправляет товар, количество или цену и сохраняет.
6. Удаляет позицию из активного отчёта через иконку корзины и подтверждение.
7. При необходимости раскрывает список исключённых товаров и восстанавливает позицию.

## Роли

- Владелец или связанный Telegram-пользователь — просматривает и корректирует данные своего магазина.
- Продавец — создаёт исходные продажи голосовым сообщением, но не определяет `shop_id` для WebApp.
- Сервер — валидирует сессию, ограничивает данные магазином и выполняет мутации через service role.

## UI/UX

- Заголовок страницы: «Продажи и выручка».
- Summary-блок показывает период, выручку и количество и остаётся компактным на мобильном экране.
- Фильтры периода прокручиваются горизонтально на узких экранах и не переносятся в несколько больших рядов.
- Карточка товара показывает название, количество с единицей, цену за единицу и сумму позиции.
- Справа находятся две кнопки размером не менее 44×44 px: редактирование и удаление.
- Текстовые ссылки «Исключить из отчёта» в активной карточке не используются.
- При отсутствии активных позиций показывается: «Нет активных товаров за выбранный период».
- Нижняя навигация содержит только «Отчёт», «Записи», «Продавцы».
- Telegram diagnostics не является частью основного отчёта и доступна только при `DEBUG_TELEGRAM_WEBAPP=true`.

## API и серверная логика

`getReport()`:

1. Получает server-derived `shop_id` через `requireOwner()`.
2. Выбирает `sales` магазина в полуинтервале `[start, end)`.
3. Загружает `sale_items` только по найденным `sale_id`.
4. Отделяет активные строки от `deleted_at is not null` и `status = excluded`.
5. Строит итог только по активным `processed` позициям с заполненными `price` и `total`.

После update/delete Server Action вызывает `revalidatePath("/daily-report")` и `revalidatePath("/records")`. Клиент закрывает карточку и выполняет `router.refresh()`, поэтому новые значения и итоги появляются без ручной перезагрузки.

## Таблицы БД

- `sales`: `id`, `shop_id`, `seller_id`, `voice_record_id`, `total_amount`, `status`, `created_at`.
- `sale_items`: `id`, `sale_id`, `product_name`, `quantity`, `unit`, `price`, `total`, `status`, `deleted_at`, `deleted_reason`, `deleted_previous_status`, `updated_at`.

В текущей схеме цена называется `price`, а сумма позиции — `total`. Поля `unit_price` и `total_price` не используются.

## Ошибки и edge cases

- Auth/DB error показывается как ошибка загрузки, а не как нулевой успешный отчёт.
- Если `sale` существует, но все её товары исключены, продажа остаётся в БД, итог равен нулю, UI показывает пустое состояние.
- `status = excluded` не показывается в активном списке даже при неконсистентном пустом `deleted_at`.
- Неуспешный update/delete не закрывает и не очищает карточку; пользователь видит стабильное сообщение, а техническая причина записывается в server logs.
- После успешного soft delete позиция не возвращается в активный отчёт после reload.

## Acceptance criteria

- Изменение количества и цены пересчитывает сумму позиции и общий итог.
- Изменения сохраняются после reload.
- Удаление уменьшает количество и выручку и сохраняется после reload.
- Все интерактивные элементы карточки доступны с клавиатуры и имеют понятные `aria-label`.
- Mobile layout не имеет горизонтального переполнения карточек.
- `npm run lint`, `npm run test`, `npm run build` проходят.

## Не входит в scope

Voice pipeline, STT, parser, webhook, физическое удаление `sales`, CRM, склад, касса, платежи и массовое редактирование.
