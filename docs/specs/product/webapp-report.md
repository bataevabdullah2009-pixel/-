# WebApp: отчёт по продажам

Статус: реализовано.

## Цель

Экран «Отчёт» даёт владельцу быстрый мобильный срез магазина: выручка, количество проданных товаров, число голосовых записей и объём того, что требует проверки.

Отчёт не заменяет кассу или склад.

Он показывает только данные, которые уже сохранены ботом в Supabase и доступны текущему server-derived магазину.

## Роли

1. Владелец открывает Telegram Mini App из кнопки бота или меню.
2. Владелец может открыть browser fallback только при явном server-side флаге.
3. Продавец создаёт исходные продажи голосом.
4. Сервер валидирует Telegram session или fallback context.
5. Сервер считает отчёт по `sales` и `sale_items`.

## Экран

Верхняя часть страницы:

1. Название страницы: «Отчёт».
2. Над названием показывается имя магазина.
3. Никакого маркетингового hero.
4. Никаких диагностических кнопок.
5. Основной контент виден сразу на мобильном экране.

Нижняя навигация:

1. «Отчёт».
2. «Записи».
3. «Проверка».
4. «Продавцы».

## Summary

Summary выполнен в dark premium dashboard стиле и состоит из четырёх компактных карточек:

1. `Выручка` — сумма активных processed позиций.
2. `Количество товаров` — суммарное количество активных processed позиций.
3. `Записей` — количество `sales` за период.
4. `Нужно проверить` — количество активных review-позиций.

Карточки не должны занимать половину экрана.

На мобильном экране они раскладываются в две колонки.

Цвет warning используется только для review-счётчика, если он больше нуля.

## Фильтры периода

Доступны:

1. Сегодня.
2. Вчера.
3. Неделя.
4. Месяц.
5. Год.
6. Выбор даты.

Фильтры компактные.

На мобильном экране preset-кнопки идут горизонтально и могут прокручиваться.

Выбор даты не должен создавать второй большой экранный блок.

Периоды считаются по `Europe/Moscow`.

## Топ товаров

Перед топом отчёт показывает мини-график продаж по дням.

Блок «Топ товаров» показывает агрегированные позиции из `summary.rows`.

Сортировка — по выручке по убыванию.

В строке показаны:

1. Название товара.
2. Суммарное количество и единица.
3. Выручка по товару.

В топ попадают только позиции, которые входят в выручку.

Review, cancelled, failed и deleted rows не попадают в топ.

## Последние продажи

Блок «Последние продажи» показывает активные processed позиции за выбранный период.

Каждая карточка товара показывает:

1. Название.
2. Строку вида `5 шт × 100 ₽`.
3. Сумму позиции.
4. Кнопку карандаша.
5. Кнопку корзины.

Постоянных больших кнопок снизу нет.

Форма редактирования раскрывается только после нажатия карандаша.

Удаление раскрывается только после нажатия корзины.

## Нужно проверить

Если есть active review items, отчёт показывает отдельный блок «Нужно проверить».

В этом блоке есть бейдж «Нужно проверить».

Report не показывает постоянные кнопки:

1. «Подтвердить позицию».
2. «Исключить из отчёта» текстовой ссылкой.
3. «Сохранить» как постоянную кнопку.

Review-запись подтверждается или отменяется в Telegram inline-кнопками либо во вкладке WebApp «Проверка».

Редактирование review item в WebApp может сохранить название, количество и цену, но не делает запись processed.

После такого редактирования item остаётся review до явного confirm.

## Исключённые товары

Исключённые товары показываются в отдельном `details` блоке.

Они не входят в активный отчёт.

Они могут быть восстановлены.

Восстановление возвращает previous status.

Если previous status был review, восстановленная позиция не входит в выручку до подтверждения.

## API и серверная логика

`getReport(filters)`:

1. Вызывает `requireOwner()`.
2. Получает server-derived `shop_id`.
3. Строит date range.
4. Читает `sales` по `shop_id` и периоду.
5. Собирает sale IDs.
6. Читает `sale_items` только по этим sale IDs.
7. Проверяет scope через `scopeReportRows`.
8. Разделяет active и deleted/excluded items.
9. Возвращает `salesCount`.
10. Возвращает `summary`.
11. Возвращает active `items`.
12. Возвращает `deletedItems`.
13. Возвращает `error`, если auth или DB не прошли.

Ошибку нельзя показывать как успешный нулевой отчёт.

## Бизнес-логика выручки

В выручку входят только позиции:

1. `status = processed`.
2. `deleted_at is null`.
3. `price is not null`.
4. `total is not null`.
5. Количество положительное.

Не входят:

1. `needs_review`.
2. Legacy `needs_price`.
3. `cancelled` sales.
4. `failed`.
5. `excluded`.
6. Rows с `deleted_at`.
7. Rows без цены.

## Ошибки и edge cases

1. Auth error показывает понятное сообщение.
2. DB error показывает понятное сообщение.
3. Если sale есть, но все items deleted, отчёт показывает пустое active state.
4. Legacy `status = excluded` без `deleted_at` не попадает в active list.
5. Если review items есть, они не смешиваются с active sales.
6. Если summary пустой, top products показывает empty state.
7. Long product name переносится внутри карточки.
8. Кнопки остаются touch-friendly.
9. Period links не переносят `mutation` и `message`.

## Acceptance criteria

1. Уверенная processed запись видна в выручке.
2. Review запись не меняет выручку.
3. Review блок есть только при review items.
4. Summary показывает 4 метрики.
5. Фильтры не занимают половину мобильного экрана.
6. Карточки товара имеют только карандаш и корзину в обычном режиме.
7. Delete уменьшает выручку и количество.
8. Update processed item пересчитывает item total и report total.
9. Update review item сохраняет поля, но выручка меняется только после confirm.
10. Reload показывает данные из Supabase.
11. Diagnostics не видны обычному production-пользователю.
12. `npm run lint`, `npm run test`, `npm run build`, `npm run web:build` проходят перед релизом.

## Не входит в scope

1. CRM.
2. Склад.
3. Касса.
4. Оплаты.
5. Клиентская база.
6. Массовое редактирование.
7. Изменение STT или LLM parser.
