# Готовность к production

Текущий MVP пригоден для учебной демонстрации, но не для реальных коммерческих данных. Ниже перечислены блокирующие требования; их реализация находится в backlog и не должна незаметно расширять MVP.

## 1. Авторизация владельца

- **Зачем:** административные страницы и мутации содержат данные продаж.
- **Текущий статус:** серверные действия используют service role без подтверждения личности владельца.
- **Риск:** любой получивший доступ к Web App может читать или менять демонстрационные данные.
- **План:** проверять Telegram Mini App init data на сервере, создать владельца и серверную сессию, записывать actor в аудит.
- **Готово, когда:** запрос без валидной сессии отклонён, а владелец работает только со своим магазином.

## 2. Изоляция данных по `shop_id`

- **Зачем:** данные разных магазинов не должны пересекаться.
- **Текущий статус:** `shop_id` есть в схеме, но не ограничивает все чтения и мутации.
- **Риск:** утечка, изменение или сброс данных другого магазина.
- **План:** выводить `shop_id` из сессии, добавить RLS и обязательные серверные фильтры, написать негативные тесты.
- **Готово, когда:** пользователь магазина A не может прочитать или изменить ни одну строку магазина B.

## 3. Ограничение доступа к Web App

- **Зачем:** URL панели не должен давать административный доступ сам по себе.
- **Текущий статус:** страницы доступны в демонстрационном режиме без production-gate.
- **Риск:** просмотр отчётов и запуск опасных действий посторонним пользователем.
- **План:** проверять сессию на входе и в каждой server action; запрещать прямой доступ и просроченные сессии.
- **Готово, когда:** неавторизованный браузер получает отказ, а клиентский параметр не может повысить права.

## 4. Идемпотентность Telegram updates

- **Зачем:** Telegram повторно доставляет update при таймауте или сетевой ошибке.
- **Текущий статус:** `telegram_message_id` не имеет обязательного уникального ограничения.
- **Риск:** двойная продажа и завышенная выручка.
- **План:** хранить уникальный ключ update, резервировать его до обработки и возвращать результат первого запуска.
- **Готово, когда:** повтор одного update не создаёт новые `voice_records`, `sales` или `sale_items`.

## 5. Транзакционное сохранение `sale` + `sale_items`

- **Зачем:** связанные строки образуют одну бизнес-операцию.
- **Текущий статус:** таблицы записываются последовательными запросами.
- **Риск:** частичная продажа, неверный итог и противоречивый аудит.
- **План:** перенести запись в SQL/RPC-транзакцию с единым commit/rollback.
- **Готово, когда:** искусственный сбой любого шага не оставляет частичных бизнес-данных.

## 6. Политика хранения аудио

- **Зачем:** аудио и расшифровки могут содержать персональные или коммерческие данные.
- **Текущий статус:** приватный bucket есть, формального срока хранения и удаления нет.
- **Риск:** избыточное хранение, рост стоимости и невозможность выполнить запрос на удаление.
- **План:** определить сроки по типам данных, lifecycle-задачу, аудит удаления и исключения для расследований.
- **Готово, когда:** политика утверждена, автоматизирована и проверена на тестовом объекте.

## 7. Резервное копирование Supabase

- **Зачем:** ошибка оператора или сбой может уничтожить БД и аудио.
- **Текущий статус:** процедура backup/restore не проверена.
- **Риск:** необратимая потеря продаж, связей и исходных файлов.
- **План:** выбрать возможности тарифа Supabase, отдельно учитывать Storage, задать RPO/RTO и проводить тест восстановления.
- **Готово, когда:** документированный restore поднимает согласованную тестовую копию в пределах RPO/RTO.

## 8. Monitoring и alerting

- **Зачем:** отказ webhook, STT, LLM или Supabase должен обнаруживаться до жалобы пользователя.
- **Текущий статус:** есть структурные журналы без метрик и оповещений.
- **Риск:** длительная потеря продаж и скрытый рост `failed`/`needs_review`.
- **План:** добавить correlation ID, метрики этапов, error tracking и алерты с runbook.
- **Готово, когда:** контролируемый сбой вызывает оповещение и связывается с конкретным update.

## 9. GitHub Actions

- **Зачем:** обязательные проверки должны выполняться одинаково для каждого изменения.
- **Текущий статус:** CI workflow отсутствует.
- **Риск:** в основную ветку попадает код с ошибкой lint, тестов, сборки или секретом.
- **План:** workflow для install, lint, test, build, проверки миграций, ссылок и запрещённых env-файлов.
- **Готово, когда:** обязательный protected check проходит на чистом runner и блокирует ошибочный PR.

## 10. Интеграционные и браузерные тесты

- **Зачем:** unit tests не проверяют связи webhook → БД → отчёт → ручная правка.
- **Текущий статус:** есть Vitest, но нет временной Supabase и browser E2E.
- **Риск:** отдельные модули проходят тесты, а пользовательский сценарий не работает.
- **План:** добавить изолированную тестовую БД, моки внешних AI-сервисов и Playwright для мобильного Web App.
- **Готово, когда:** критический и негативный сценарии выполняются в CI без production-секретов.
