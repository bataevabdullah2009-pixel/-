# Редактирование и удаление товарной позиции

Статус: реализовано.

## Цель

Позволить владельцу исправить результат распознавания непосредственно в карточке товара, не переходя на отдельную страницу и не затрагивая голосовой pipeline.

## Пользовательские сценарии

### Редактирование

1. Пользователь нажимает карандаш.
2. Карточка раскрывает поля «Товар», «Количество», «Цена, ₽».
3. Пользователь меняет одно или несколько значений.
4. «Сохранить» блокируется и показывает «Сохраняем…».
5. После подтверждённого update карточка закрывается, сумма и общий отчёт обновляются.
6. «Отмена» сбрасывает несохранённый ввод к значениям из БД.

### Удаление

1. Пользователь нажимает корзину.
2. Внутри карточки появляется вопрос «Удалить товар из отчёта?».
3. «Удалить» выполняет soft delete, «Отмена» закрывает подтверждение.
4. После успеха карточка исчезает из активного списка и итоги пересчитываются.

## Роли

- Пользователь WebApp с валидным контекстом магазина — инициирует действие.
- Server Action — валидирует форму и возвращает безопасное состояние `success/error`.
- Records API — повторно проверяет принадлежность `sale` текущему магазину и выполняет Supabase update.

## UI/UX

- Обычная карточка не содержит постоянно раскрытую форму.
- Поля имеют высоту не менее 44 px и поддерживают `inputMode="decimal"`.
- Кнопки «Сохранить», «Отмена», «Удалить» имеют высоту не менее 44 px.
- При ошибке форма остаётся открытой, введённые значения не очищаются.
- Ошибка выводится внутри той карточки, к которой относится.
- В review-позиции успешное сохранение валидных полей переводит товар в «Готово».

## API и серверная логика

`updateSaleItemAction(previousState, formData)`:

- проверяет `itemId`, непустое название, положительные quantity/price;
- вызывает `updateSaleItem`;
- логирует technical reason, если API вернул `ok: false`;
- возвращает клиенту только стабильное русскоязычное сообщение;
- при успехе инвалидирует отчёт и журнал.

`updateSaleItem`:

- читает текущую строку;
- запрещает редактирование soft-deleted позиции;
- проверяет `sale_id -> sales.shop_id`;
- нормализует название и единицу;
- обновляет `product_name`, `quantity`, `price`, `total`, `status`, `confidence`, `updated_at`;
- читает изменённую строку через `.select(...).single()`;
- пересчитывает `sales.total_amount` и статусы;
- записывает audit log best-effort.

`excludeSaleItem`:

- устанавливает `status = excluded`;
- заполняет `deleted_at`, `deleted_reason = excluded_by_owner`, `deleted_previous_status`, `updated_at`;
- не удаляет `sale`;
- пересчитывает родительскую продажу.

## Таблицы БД

Главная таблица — `sale_items`. Связь с магазином устанавливается только через `sale_items.sale_id -> sales.id -> sales.shop_id`.

Обязательные поля update:

- `product_name`;
- `quantity`;
- `price`;
- `total`;
- `status`;
- `updated_at`.

Поля soft delete:

- `deleted_at`;
- `deleted_reason`;
- `deleted_previous_status`.

## Ошибки и edge cases

- Пустое название, quantity ≤ 0 или price ≤ 0 отклоняются до запроса к БД.
- Отсутствующая строка или чужой `sale_id` не изменяются.
- Повторное удаление идемпотентно возвращает успешное пользовательское сообщение.
- Ошибка пересчёта не маскируется как успех.
- Audit log failure логируется отдельно и не отменяет уже подтверждённую основную мутацию.
- Legacy `excluded` без `deleted_at` не попадает в активный UI.

## Acceptance criteria

- Сохраняются название, количество и цена.
- `total = quantity × price` с округлением до двух знаков.
- После update/delete отчёт обновляется без ручного reload.
- После reload состояние совпадает с БД.
- Отмена не выполняет сетевой запрос и возвращает исходные значения.
- Удаление одного item не удаляет `sales`.

## Не входит в scope

Изменение единицы измерения отдельным полем, массовые операции, изменение времени продажи, восстановление всей продажи, редактирование voice/STT/parser данных.
