# Глобальная спецификация проекта

Статус на 2026-07-05: этот документ является главным системным документом проекта. Если краткие обзорные документы расходятся с ним, актуальным источником считается этот файл вместе с фактическим кодом и миграциями Supabase.

## 1. Назначение проекта

`Голосовой журнал продаж` - коммерческий Telegram-бот и Telegram WebApp для магазина, где продавец фиксирует продажу голосом, а система превращает речь в структурированные позиции продажи.

Проект решает четыре задачи:

1. Быстро принять продажу от продавца без ручного заполнения таблиц.
2. Сохранить исходную голосовую запись, распознанный текст, продажу и товары в Supabase.
3. Отделить уверенные позиции от сомнительных, чтобы выручка не загрязнялась неполными данными.
4. Показать владельцу WebApp с отчётом, проверкой, журналом записей и продавцами.

Главная бизнес-метрика системы - корректная выручка. Любое поведение бота, WebApp и базы данных должно защищать расчёт выручки.

## 2. Роли

1. **Продавец** отправляет голосовые продажи в Telegram и может открывать WebApp магазина.
2. **Владелец** просматривает WebApp магазина, проверяет сомнительные позиции и управляет позициями продажи.
3. **Telegram-бот** принимает `/start`, голосовые сообщения и callback-кнопки проверки.
4. **WebApp** показывает отчёт, проверку, записи и продавцов; все бизнес-мутации выполняет через серверный контекст.
5. **Supabase** хранит магазины, продавцов, каталог товаров, голосовые записи, продажи, позиции продажи и audit logs.
6. **STT/LLM** участвуют только в распознавании и нормализации текста; финальная защита перед сохранением остаётся детерминированной.

Owner binding хранится в таблице `owners`: активный владелец определяется по Telegram user id, получает доступ только к своему магазину и может создать/привязать seller-контекст только внутри этого же магазина. Владелец или продавец без активной привязки не получает доступ к данным магазина. Создание продавца владельцем не является мульти-магазинным переключателем и не даёт прав на чужие магазины.

## 3. Системные границы

Основные части проекта:

1. `apps/bot` - Telegram bot, voice pipeline, callback handlers.
2. `apps/web` - Next.js Telegram WebApp, route handlers, Server Components и Server Actions.
3. `packages/shared` - общие типы, схемы, парсер, правила evidence и утилиты отчёта.
4. `supabase/migrations` - схема БД, RLS/grants, RPC и soft delete.
5. `docs` - документация, спецификации, правила, планы и feature-документы.
6. `scripts` - служебные команды Telegram webhook.

Каноническая архитектура хранится в [docs/architecture/architecture.md](../architecture/architecture.md).

Серверная граница:

1. `POST /api/telegram/webhook` принимает Telegram updates и передаёт их в общий bot processor.
2. `POST /api/auth/telegram` bootstrap-ит WebApp session по Telegram initData или явно разрешённому fallback.
3. Отчёт, проверка, записи и продавцы не имеют отдельного публичного business REST API.
4. WebApp читает данные через Server Components.
5. Бизнес-мутации WebApp выполняются через Server Actions.
6. Клиент не пишет напрямую в бизнес-таблицы Supabase.
7. Service role используется только server-side.

## 4. Голосовой сценарий

Базовый путь продавца:

1. Продавец открывает Telegram-бота.
2. `/start` проверяет привязку к магазину и показывает вход в WebApp.
3. Продавец отправляет голосовое сообщение с продажей.
4. Бот определяет продавца по Telegram user id.
5. Бот скачивает аудио из Telegram.
6. Бот подготавливает аудио для STT.
7. Бот сохраняет аудио в Storage best effort.
8. STT возвращает русскую расшифровку.
9. LLM очищает текст и пытается выделить позиции.
10. Детерминированный слой evidence сверяет позиции с расшифровкой.
11. Резервный парсер разделяет склеенные фразы на отдельные `sale_items`.
12. Supabase сохраняет `voice_records`, `sales` и `sale_items`.
13. Бот читает результат обратно, чтобы не отправить ложный успех.
14. Уверенная запись получает обычный ответ об успешном сохранении.
15. Сомнительная запись получает сообщение проверки с двумя кнопками.
16. WebApp показывает обновлённые данные в отчёте, проверке, записях и продавцах.

## 5. Поток Telegram-бота

### `/start`

1. Проверяет Telegram user id.
2. Резолвит активного продавца или допустимый demo/fallback-контекст.
3. Показывает кнопку открытия WebApp.
4. Может использовать reply/menu button для входа в отчёт.
5. Не подтверждает и не отменяет продажи.

### Голосовое сообщение

1. Невалидный или неизвестный продавец не может сохранить продажу.
2. Аудио загружается и конвертируется best effort.
3. Сбой загрузки аудио в Storage не должен блокировать сохранение продажи.
4. Сбой STT до сохранения продажи фиксируется как диагностируемый `failed`.
5. Невалидный JSON от LLM не должен терять запись; используется детерминированный fallback.
6. Сохранение в Supabase должно завершиться read-back проверкой.
7. Если количество сохранённых позиций не совпало с ожидаемым, успех пользователю не отправляется.

### Аудио и Storage

1. Основной bucket для аудио - `voice-records`.
2. Загрузка аудио в Storage выполняется best effort.
3. Сбой Storage не блокирует сохранение продажи, если STT и бизнес-данные можно сохранить.
4. Ссылка на аудио в журнале создаётся server-side как signed URL.
5. Отсутствие аудио не должно удалять продажу, портить выручку или скрывать запись из журнала.
6. Путь или URL аудио не является источником прав доступа.

### Ответы бота

1. Уверенная запись отвечает сообщением `✅ Запись сохранена: ...`.
2. Сомнительная запись отвечает предупреждением и показывает только две inline-кнопки: `✅ Подтвердить` и `❌ Отмена`.
3. В сообщении проверки не должно быть кнопки `Открыть отчёт`.
4. `Открыть отчёт` допустим в `/start`, reply keyboard или menu button.

### Callback

Новые callback-данные:

```text
confirm:<sale_id>
cancel:<sale_id>
```

Устаревший формат `voice_sale_review:<action>:<sale_id>` может приниматься только для старых сообщений.

Callbacks должны быть идемпотентными: повторное нажатие не создаёт дубли, не ломает статусы и возвращает понятный результат.

## 6. Поток WebApp

1. WebApp запускается как Telegram Mini App или явно разрешённый fallback в браузере.
2. Клиент передаёт Telegram initData.
3. Сервер проверяет initData через токен бота.
4. Сервер выводит principal владельца или продавца.
5. Сервер определяет магазин.
6. Серверные компоненты читают данные только для текущего магазина.
7. Server Actions повторно проверяют доступ к магазину перед мутацией.
8. Клиентский `shop_id` не является источником прав.
9. Service role key никогда не попадает в браузерный bundle.
10. Ошибки auth/БД показываются как ошибки, а не как пустой отчёт.

Owner/fallback правила:

1. Активный seller получает доступ только к своему `seller.shop_id`.
2. Активный owner получает доступ только к своему `owners.shop_id`.
3. Owner binding может создать seller-контекст только в магазине владельца.
4. Fallback в браузере разрешён только при `ALLOW_WEBAPP_FALLBACK=true`.
5. Fallback требует `DEFAULT_SHOP_ID` и `DEFAULT_SELLER_ID`.
6. Сервер обязан проверить, что fallback seller активен и принадлежит `DEFAULT_SHOP_ID`.
7. Demo/fallback не должен обходить серверную проверку магазина.

## 7. Вкладки WebApp

Нижняя навигация содержит четыре вкладки:

1. `Отчёт` - сводка магазина, выручка, количество товаров, записи, проверки, топ товаров, динамика по дням и продажи за период.
2. `Проверка` - активные сомнительные позиции `needs_review`, действия подтверждения/отмены, массовое подтверждение и ручное исправление.
3. `Записи` - журнал голосовых продаж с датой, продавцом, распознанным текстом, статусом, суммой, товарами, поиском, фильтром продавца и аудио при наличии.
4. `Продавцы` - список продавцов, активность, количество записей и выручка за выбранный период.

Маршрут `/` ведёт к отчёту. Диагностический `/debug-telegram` доступен только в разработке или при `DEBUG_TELEGRAM_WEBAPP=true`.

## 8. Статусы записей

Для `voice_records.status` и `sales.status` используются:

1. `pending` - техническое начальное состояние, не является завершённым пользовательским результатом.
2. `processed` - запись уверенно обработана или подтверждена; выручка считается по активным обработанным позициям.
3. `needs_review` - запись сохранена, но содержит позиции, требующие проверки.
4. `cancelled` - пользователь отменил запись; родительская продажа не входит в выручку.
5. `failed` - технический сбой обработки; запись не входит в выручку.

Важно: родительский `needs_review` не блокирует выручку уже обработанной соседней позиции. Выручка определяется на уровне `sale_items`, если родительская продажа не `cancelled` и не `failed`.

## 9. Статусы товаров и позиций

Для `sale_items.status` используются:

1. `processed` - активная позиция может входить в выручку.
2. `needs_review` - позиция требует проверки и не входит в выручку.
3. `needs_price` - legacy-состояние, трактуется как проверка и не используется для новых неполных позиций.
4. `failed` - позиция не входит в выручку.
5. `excluded` - мягко удалённая позиция, не входит в активный отчёт.

Каталог `products` не имеет отдельного lifecycle-статуса в текущей схеме. Для пользовательского сценария "товар" означает строку `sale_items`, созданную из голосовой продажи или отредактированную в WebApp.

## 10. Правила полной позиции

Позиция считается полной, если выполнены все условия:

1. Есть осмысленное название товара.
2. Есть валидное количество или вес.
3. Единица измерения нормализована: `шт`, `кг` или `г`.
4. Есть валидная цена за единицу или валидная сумма, из которой цену можно вывести.
5. `total` является положительным числом или может быть корректно пересчитан.
6. Значение не помечено как мягко удалённое.
7. Уверенность распознавания достаточна либо позиция вручную исправлена пользователем.

Полная уверенная позиция получает `processed`. После ручного исправления в WebApp валидная позиция также становится `processed`, даже если родительская продажа остаётся `needs_review` из-за соседних неполных позиций.

## 11. Правила сомнительной позиции

Позиция считается сомнительной, если есть хотя бы одно условие:

1. Нет отдельного осмысленного товара.
2. Нет количества или веса.
3. Нет цены и нет суммы, из которой цену можно вывести.
4. Сумма невалидна.
5. Уверенность распознавания низкая.
6. LLM вернул склеенную или неоднозначную фразу.
7. Резервный парсер выделил неполный остаток.
8. Позиция пришла из legacy-состояния `needs_price`.

Сомнительная позиция получает `needs_review`, показывается во вкладке `Проверка` и не входит в выручку.

## 12. Смешанная корзина

Одна голосовая запись может содержать полные и неполные позиции.

Пример:

```text
Хлеб 3 штуки по 50 рублей. Корзина продуктов.
```

Ожидаемое поведение:

1. `Хлеб` сохраняется отдельной полной позицией.
2. `Корзина продуктов` сохраняется отдельной позицией на проверку.
3. Родительская продажа может остаться `needs_review`.
4. Активная позиция `processed` может войти в выручку.
5. Неполная позиция остаётся во вкладке `Проверка`.
6. Подтверждение подтверждает валидные позиции и оставляет неполные.
7. Отмена отменяет всю голосовую запись и мягко удаляет активные позиции.

## 13. Проверка

Проверка нужна, когда система не может безопасно включить все позиции в выручку.

Проверка доступна двумя путями:

1. Inline-кнопки Telegram под сообщением сомнительной записи.
2. Вкладка WebApp `Проверка`.

Оба пути используют одинаковые доменные правила:

1. Доступ проверяется на сервере.
2. Проверяются активные `sale_items`.
3. Подтверждаются только полные позиции.
4. Неполные позиции не включаются в выручку.
5. Отмена исключает родительскую запись из выручки.
6. Повторные действия не должны создавать дубли или ложные ошибки.
7. Массовое подтверждение в WebApp обрабатывает только доступные текущему магазину продажи и не должно подтверждать неполные позиции.

## 14. Подтверждение

Подтверждение выполняется для `sales.status = needs_review` или совместимых исторических состояний.

Правила:

1. Найти продажу в магазине текущего продавца/владельца.
2. Найти активные позиции с `deleted_at is null`.
3. Проверить каждую позицию отдельно.
4. Если нет ни одной полной позиции, не менять данные и вернуть сообщение `Не удалось подтвердить: нет ни одной полной позиции.`
5. Полные активные позиции перевести в `processed`.
6. Неполные активные позиции оставить `needs_review`.
7. Пересчитать `sales.total_amount` по активным обработанным позициям.
8. Если активных сомнительных позиций больше нет, поставить `sales.status = processed` и `voice_records.status = processed`.
9. Если сомнительные позиции остались, родительская запись может остаться `needs_review`, но обработанные позиции уже входят в выручку.
10. Успешный ответ должен быть вида `✅ Подтверждено: N позиций, сумма X ₽`.
11. Повторное подтверждение уже обработанной записи возвращает успешный идемпотентный результат.

## 15. Отмена

Отмена означает, что пользователь отклоняет всю сомнительную голосовую запись.

Правила:

1. Найти продажу в магазине текущего продавца/владельца.
2. Активные позиции этой продажи мягко удалить.
3. Для каждой позиции сохранить `deleted_at`, `deleted_reason` и `deleted_previous_status`.
4. Поставить `sales.status = cancelled`.
5. Поставить `voice_records.status = cancelled`.
6. Поставить `sales.total_amount = 0`.
7. Исключить запись и её позиции из выручки.
8. Повторная отмена уже отменённой записи возвращает идемпотентный результат.

Физическое удаление позиций продажи не используется в пользовательском сценарии.

## 16. Редактирование и удаление позиций

Редактирование в WebApp:

1. Открывается через иконку карандаша.
2. Показывает поля: товар, количество, единица, цена.
3. Поддерживает единицы `шт`, `кг`, `г`.
4. Для граммов сумма считается как доля килограмма от цены за кг.
5. Сохраняет изменения через Server Action.
6. Проверяет доступ к магазину на сервере.
7. Обновляет реальную строку `sale_items`.
8. Пересчитывает `sale_items.total`.
9. Ставит валидную позицию в `processed`.
10. Пересчитывает `sales.total_amount`.

Удаление:

1. Открывается через иконку корзины.
2. Показывает подтверждение `Удалить товар из отчёта?`.
3. Выполняет мягкое удаление.
4. Убирает позицию из активного отчёта.
5. Сохраняется после перезагрузки страницы.

Восстановление:

1. Доступно только для мягко удалённых позиций текущего магазина.
2. Возвращает позицию в предыдущий статус из `deleted_previous_status`.
3. Очищает `deleted_at`, `deleted_reason` и `deleted_previous_status`.
4. Пересчитывает родительскую продажу и отчёт.
5. Не восстанавливает физически удалённые строки.

Сброс дня:

1. Выполняется через серверное действие.
2. Ограничен выбранным дневным диапазоном.
3. Мягко удаляет активные позиции дня с причиной `day_reset`.
4. Не удаляет `sales` и `voice_records`.
5. Пересчитывает выручку после исключения позиций.

## 17. Правила выручки

В выручку входит только позиция, которая одновременно удовлетворяет условиям:

1. Родительская продажа принадлежит текущему магазину.
2. Родительская продажа не `cancelled`.
3. Родительская продажа не `failed`.
4. `sale_items.status = processed`.
5. `sale_items.deleted_at is null`.
6. `sale_items.total` существует и является валидным положительным числом.
7. Количество или вес валидны.
8. Цена за единицу валидна или может быть выведена из `total / quantity`.

В выручку не входят:

1. Родительская продажа `cancelled`.
2. Родительская продажа `failed`.
3. Позиция `needs_review`.
4. Позиция `needs_price`.
5. Позиция `failed`.
6. Позиция `excluded`.
7. Любая позиция с `deleted_at`.

## 18. Аналитика

Аналитика WebApp строится из тех же правил выручки:

1. Сводная выручка за период.
2. Количество проданных товаров.
3. Количество голосовых записей.
4. Количество активных позиций на проверку.
5. Топ товаров.
6. Динамика по дням.
7. Продажи за выбранный период.
8. Статистика продавцов за период.

Фильтр периода поддерживает сегодня, вчера, неделю, месяц и пользовательскую дату. Даты считаются в проектной зоне `Europe/Moscow`.

## 19. Объём MVP

В MVP входит:

1. Telegram `/start`.
2. Приём голосовых сообщений.
3. STT и LLM cleanup/parser.
4. Детерминированный fallback для склеенных и неполных позиций.
5. Сохранение `voice_records`, `sales`, `sale_items`.
6. Подтверждение и отмена сомнительных продаж в Telegram.
7. Проверка сомнительных позиций в WebApp.
8. Отчёт по выручке.
9. Журнал записей.
10. Список продавцов и их показатели.
11. Редактирование, мягкое удаление и восстановление позиций.
12. Сброс выручки выбранного дня через soft delete позиций.
13. Серверная авторизация Telegram WebApp.
14. Owner binding и создание seller-контекста только в магазине владельца.
15. Fallback-режим только при явном флаге окружения и проверке seller/shop.
16. Поиск и фильтр продавца в журнале записей.
17. Массовое подтверждение проверяемых продаж в WebApp.
18. Документация текущего поведения.

## 20. Что не входит

Не входит в текущий объём:

1. Складской учёт и списание остатков.
2. Бухгалтерская отчётность.
3. Платежи и кассовая интеграция.
4. Мульти-магазинный переключатель в UI.
5. Публичные клиентские записи в бизнес-таблицы.
6. Физическое удаление позиций продажи как пользовательский путь.
7. Полная замена STT или LLM-провайдера.
8. Полное переписывание Telegram bot logic.
9. Полное переписывание WebApp logic.
10. Изменение Supabase schema без отдельной миграционной задачи.
11. Экспорт CSV/XLSX, если он явно не реализован.
12. Продвинутая аналитика уверенности парсера за пределами текущих диагностик.

## 21. Критерии приёмки

Проект считается соответствующим спецификации, если:

1. Уверенная голосовая продажа создаёт `processed` запись и попадает в отчёт.
2. Сомнительная голосовая продажа создаёт `needs_review` и не попадает в выручку до решения.
3. Сообщение проверки Telegram имеет ровно две кнопки: `✅ Подтвердить` и `❌ Отмена`.
4. В сообщении проверки нет кнопки `Открыть отчёт`.
5. Webhook принимает `message` и `callback_query`.
6. Подтверждение смешанной корзины подтверждает валидные позиции и оставляет неполные на проверке.
7. Подтверждение без полной позиции возвращает понятную ошибку и не меняет данные.
8. Отмена переводит запись в `cancelled` и мягко удаляет активные позиции.
9. Отчёт считает выручку по активным `sale_items.status = processed`.
10. Родительский `needs_review` не блокирует обработанную соседнюю позицию.
11. Редактирование валидной позиции сохраняется в Supabase и пересчитывает суммы.
12. Удаление позиции сохраняется после перезагрузки и исключает её из отчёта.
13. WebApp имеет вкладки `Отчёт`, `Проверка`, `Записи`, `Продавцы`.
14. Ошибки авторизации или БД не маскируются пустыми данными.
15. Owner получает доступ только к своему магазину и не может создать seller в чужом магазине.
16. Fallback работает только при явном флаге и совпадающих `DEFAULT_SHOP_ID` / `DEFAULT_SELLER_ID`.
17. Публичного business REST API для отчёта и мутаций нет; WebApp использует Server Components и Server Actions.
18. Сбой Storage не блокирует сохранение продажи, если бизнес-данные сохранены.
19. Восстановление soft-deleted позиции пересчитывает отчёт.
20. Сброс дня мягко исключает позиции выбранного дня и не удаляет продажи.
21. Массовое подтверждение не подтверждает неполные позиции и не выходит за границы магазина.
22. Логи не содержат raw initData, токены и service role key.
23. Документация ссылается на этот документ как на главный системный источник.

## 22. Тестовые сценарии

Минимальные сценарии для проверки продукта:

1. `Сникерс пять штук по сто рублей` создаёт одну позицию `processed` с суммой `500`.
2. `Буханка хлеба пять штук по сто рублей. Сникерс три штуки по двести рублей.` создаёт две позиции и сумму `1100`.
3. `Сникерс, 3 штуки по 200 рублей. Буханка хлеба, 5 штук по 50 рублей.` разделяется на две позиции и сумму `850`.
4. `Корзина продуктов.` создаёт позицию на проверку и не подтверждается без полной позиции.
5. Смешанная запись `Хлеб 3 штуки по 50 рублей. Корзина продуктов.` подтверждает хлеб и оставляет неполную позицию в `Проверке`.
6. Повторное `✅ Подтвердить` по уже подтверждённой записи не создаёт дубли.
7. `❌ Отмена` переводит запись в `cancelled`, мягко удаляет активные позиции и убирает выручку.
8. Редактирование товара через карандаш сохраняет товар, количество, единицу и цену после перезагрузки.
9. Удаление товара через корзину сохраняет soft delete после перезагрузки.
10. Отчёт за сегодня показывает только активные обработанные позиции текущего магазина.
11. Вкладка `Проверка` показывает только активные позиции `needs_review`.
12. Вкладка `Записи` показывает исходный текст и раскрытие товаров.
13. Вкладка `Продавцы` считает записи и выручку за выбранный период.
14. WebApp без валидного Telegram initData и без разрешённого fallback показывает ошибку доступа.
15. `/debug-telegram` в production недоступен без `DEBUG_TELEGRAM_WEBAPP=true`.
16. Fallback с несовпадающим `DEFAULT_SELLER_ID` / `DEFAULT_SHOP_ID` возвращает ошибку доступа.
17. Owner без seller-привязки получает seller-контекст только в своём магазине.
18. Восстановление удалённой позиции возвращает её в отчёт, если previous status допускает выручку.
19. Сброс дня исключает активные позиции выбранного дня и сохраняет родительские продажи.
20. Поиск и фильтр продавца в `Записях` не показывают данные чужого магазина.
21. Массовое подтверждение в `Проверке` подтверждает только валидные позиции.

## 23. Ошибки и логирование

1. Ошибки авторизации показываются пользователю как ошибки доступа.
2. Ошибки БД показываются как ошибки загрузки или мутации, а не как успешные пустые данные.
3. Ошибки STT/LLM/parser имеют диагностируемую стадию.
4. Невосстановимые ошибки до сохранения продажи могут создавать `failed` запись.
5. Recoverable parser issues должны создавать проверяемые позиции, а не терять продажу.
6. Audit log failures являются best effort и не блокируют пользовательскую мутацию.
7. Revalidation failures после успешной мутации не должны превращать успешную мутацию в ложную ошибку статуса.
8. Логи могут содержать только безопасную metadata: mode, ids, stage, counts, boolean-флаги конфигурации и коды ошибок.
9. Логи не должны содержать raw initData, Telegram token, webhook secret, Supabase service role key, STT/LLM keys и пользовательские секреты.
10. Debug route в production доступен только при явном `DEBUG_TELEGRAM_WEBAPP=true`.
11. Voice lifecycle логируется отдельными событиями от `VOICE_RECEIVED` до `VOICE_PROCESSING_COMPLETED` или `VOICE_PROCESSING_FAILED`; failure содержит stage, безопасную metadata Telegram, error name/message/code, HTTP status, ограниченный response body и network cause.
12. Полный transcript и parser JSON не выводятся в runtime lifecycle logs. Их хранение в защищённой БД/audit log регулируется отдельно.
13. Если ошибка произошла на `seller_resolve`, `failed`-запись может быть недоступна для сохранения: первичным источником диагностики остаётся server log.

## 24. Release gate

Проект можно считать готовым к production только если выполнены условия:

1. Заданы обязательные env для Telegram, WebApp, Supabase и STT/LLM.
2. Supabase migrations применены.
3. Таблицы, статусы, soft delete columns, RPC `save_voice_sale`, RLS/grants и bucket `voice-records` соответствуют specs.
4. `SUPABASE_SERVICE_ROLE_KEY` доступен только server-side.
5. Локальные проверки качества прошли: `lint`, `test`, `build`, `web:build`.
6. Telegram webhook установлен на production URL.
7. `telegram:webhook-info` показывает `message` и `callback_query`.
8. WebApp открывается из Telegram Mini App с валидным initData.
9. Production smoke проверяет уверенную, сомнительную, смешанную и отменённую продажу.
10. Проверены подтверждение и отмена в Telegram.
11. Проверены проверка, редактирование, удаление, восстановление, сброс дня, записи и продавцы в WebApp.
12. Проверены Vercel/Supabase logs: нет raw initData и секретов.
13. Ошибки доступа и БД отображаются явно.
14. Fallback выключен в production или явно согласован и проверен.
15. Production Supabase project имеет рабочий статус и отвечает на API-запросы; состояние `INACTIVE` блокирует и voice pipeline, и WebApp, даже если build и webhook исправны.
16. External smoke проверяет реальные STT/LLM endpoints, production schema/RPC/bucket, WebApp auth/pages/assets и Telegram webhook/menu URL. Синтетический initData не заменяет ручное открытие Mini App на поддерживаемых Telegram clients.
