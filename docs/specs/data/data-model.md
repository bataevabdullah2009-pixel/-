# Модель данных

```text
shops 1 ── * sellers
  │          │
  ├── * products
  ├── * voice_records 1 ── 0..1 sales 1 ── * sale_items
  └── * sales

audit_logs → необязательный shop/seller + details JSON
объект Storage ← voice_records.audio_path
```

## Создание

Голосовая обработка находит или создаёт магазин и продавца, загружает аудио, затем создаёт голосовую запись, продажу, позиции и события аудита. Цена берётся из речи; при отсутствии может использоваться цена активного совпавшего товара.

Операции выполняются последовательно и пока не объединены транзакцией. При промежуточной ошибке возможны частичные данные.

## Чтение

Записи выбираются по `sales.created_at`; отчёт читает связанные позиции. Подписанная ссылка Storage действует 30 минут. Группировка отчёта не меняет исходные строки.

## Изменение

Владелец меняет только итоговую позицию продажи. Совпадение с товаром может установить `product_id`, каноническое имя и единицу. Пересчитываются итог, статус позиции, итог и статус продажи, а также статус голосовой записи. Аудит хранит состояние до и после.

## Исключение и сброс

Интерфейс не выполняет SQL `DELETE`. Поля `deleted_at`, `deleted_reason` и `deleted_previous_status` исключают позицию. Сброс выбирает продажи дня и устанавливает метаданные всем активным позициям. Голосовая запись, продажа, расшифровка, JSON парсера и аудио сохраняются.

## Восстановление

Восстановление возвращает `deleted_previous_status`, очищает поля удаления и пересчитывает продажу. Позиция с прежним `needs_price` возвращается в блок проверки, а не автоматически в выручку.

## Физическое хранение

Автоматическая очистка аудио и записей не реализована. Физическое удаление требует отдельной политики сроков хранения, резервного копирования и активного плана.
