# Панель владельца

Статус: реализовано в Telegram WebApp через Next.js App Router.

Панель владельца - это пользовательский WebApp, который владелец или разрешённый продавец открывает из Telegram. Панель не является публичной аналитической страницей: все чтения и изменения проходят через серверный контекст, полученный из Telegram initData, привязки owner/seller или явно включённого fallback.

## Цели экрана

1. Показать выручку магазина за выбранный период.
2. Показать количество проданных товаров.
3. Показать количество голосовых записей продаж.
4. Показать сколько позиций нужно проверить.
5. Дать владельцу быстрый доступ к проверке.
6. Дать владельцу журнал исходных голосовых записей.
7. Дать владельцу статистику продавцов.
8. Позволить исправить ошибочно распознанный товар.
9. Позволить исключить товар из активного отчёта без физического удаления.
10. Позволить восстановить мягко удалённый товар.
11. Позволить сбросить выручку за один день через мягкое удаление.
12. Не показывать чужой магазин даже при подмене query/form data.

## Навигация

Нижняя навигация содержит четыре пользовательских раздела:

1. `Отчёт` -> `/daily-report`.
2. `Проверка` -> `/review`.
3. `Записи` -> `/records`.
4. `Продавцы` -> `/sellers`.

Root `/` ведёт к отчёту. Диагностика не входит в основную навигацию.

## Отчёт

Owner видит:

1. Название текущего магазина.
2. Период отчёта.
3. Метрику `Выручка`.
4. Метрику `Количество товаров`.
5. Метрику `Записей`.
6. Метрику `Нужно проверить`.
7. Фильтр периода.
8. Действие обновления.
9. Мини-график продаж по дням.
10. Топ товаров.
11. Последние активные продажи.
12. Блок позиций проверки, если есть незавершённые позиции.
13. Блок исключённых товаров, если есть мягко удалённые строки.

Фильтр периода поддерживает:

1. Сегодня.
2. Вчера.
3. Последние 7 дней.
4. Месяц.
5. Выбранную дату.

## Правила выручки

Панель считает выручку на уровне позиции:

1. Родительская продажа должна быть в текущем магазине.
2. Родительская продажа не должна быть `cancelled`.
3. Родительская продажа не должна быть `failed`.
4. Позиция должна быть активной.
5. `deleted_at` позиции должен быть `null`.
6. `status` позиции должен быть `processed`.
7. `total` позиции должен быть валидным.
8. Quantity/weight должны быть валидными.

Родительская продажа со статусом `needs_review` может иметь уже обработанную соседнюю позицию. Такая позиция входит в выручку, пока неполные соседние позиции остаются в `Проверке`.

## Проверка

Вкладка `Проверка` показывает активные позиции проверки:

1. Не `processed`.
2. Не `excluded`.
3. Не мягко удалена.
4. В родительской продаже текущего магазина.
5. В периоде фильтра.

Owner может:

1. Подтвердить одну запись.
2. Отменить одну запись.
3. Подтвердить все видимые записи.
4. Отредактировать отдельную позицию.
5. Исключить отдельную позицию.

Подтверждение работает по активным позициям. Если в смешанной продаже есть одна валидная позиция и одна неполная, подтверждение переводит валидную позицию в `processed`, пересчитывает `total_amount` и оставляет неполную позицию в `needs_review`.

Если нет ни одной полной позиции, подтверждение возвращает `Не удалось подтвердить: нет ни одной полной позиции.` и не меняет строки.

## Журнал записей

Вкладка `Записи` показывает:

1. Дату и время.
2. Продавца.
3. Распознанный текст.
4. Резервный raw-текст.
5. Пользовательский статус.
6. Общую сумму.
7. Ссылку на аудио, если аудио сохранено.
8. Раскрываемый список товаров.

Журнал не подтверждает и не отменяет голосовую продажу. Для решения проверки используется вкладка `Проверка` или callback Telegram.

## Продавцы

Вкладка `Продавцы` показывает:

1. Имя продавца.
2. Активность.
3. Последнюю активность за период.
4. Количество записей за период.
5. Выручку за период.

Выручка продавца считается по тем же правилам уровня позиции, что отчёт панели. Позиции проверки, cancelled, failed и удалённые строки не входят.

## Редактирование товара

Карандаш раскрывает компактную форму:

1. `Товар`.
2. `Количество`.
3. `Единица`.
4. `Цена, ₽`.

Сохранение:

1. Проверяет сессию.
2. Проверяет, что позиция существует.
3. Проверяет родительскую продажу.
4. Проверяет доступ к магазину.
5. Валидирует название товара.
6. Валидирует количество.
7. Валидирует цену.
8. Нормализует единицу.
9. Пересчитывает `total`.
10. Ставит `status = processed`.
11. Ставит `confidence = 1`.
12. Обновляет `updated_at`.
13. Пересчитывает родительскую продажу.
14. Делает revalidate `/daily-report`, `/review`, `/records`, `/sellers`.

Если родительская продажа ещё `needs_review`, обычное редактирование не подтверждает всю голосовую запись. Оно делает готовой только исправленную строку позиции. Эта строка позиции может войти в выручку, а соседние неполные строки остаются на проверке.

## Удаление и восстановление товара

Корзина раскрывает локальное подтверждение `Удалить товар из отчёта?`.

Удаление:

1. Не удаляет строку физически.
2. Ставит `status = excluded`.
3. Ставит `deleted_at`.
4. Ставит `deleted_reason = excluded_by_owner`.
5. Сохраняет `deleted_previous_status`.
6. Пересчитывает родительскую продажу.
7. Пишет audit log по возможности.

Восстановление:

1. Доступно для мягко удалённых строк в панели деталей.
2. Возвращает предыдущий статус.
3. Очищает metadata удаления.
4. Пересчитывает родительскую продажу.
5. Обновляет экран через revalidation.

## Сброс дня

Owner может сбросить выручку за один день только для однодневного диапазона.

`resetDayRevenueAction`:

1. Принимает period/date из формы.
2. Разрешает только `today`, `yesterday`, `custom`.
3. Проверяет корректность custom даты.
4. Вычисляет диапазон в `Europe/Moscow`.
5. Ограничивает диапазон одним днём.
6. Находит продажи текущего магазина.
7. Мягко удаляет активные позиции.
8. Ставит `deleted_reason = day_reset`.
9. Пересчитывает родительские продажи.
10. Пишет audit log.
11. Возвращает читаемое сообщение.

## Ошибки

Dashboard должен показывать ошибки явно:

1. Отсутствует Telegram initData -> открыть WebApp внутри Telegram.
2. Невалидный initData -> закрыть и открыть отчёт заново.
3. Продавец не привязан -> нет доступа к магазину.
4. Продавец неактивен -> доступ отключён.
5. Магазин не найден -> магазин не найден.
6. Отсутствует Supabase admin -> серверная ошибка действия.
7. Сбой загрузки БД -> сообщение об ошибке, не пустое состояние.

## Ожидания безопасности

1. Клиент не передаёт доверенный `shop_id`.
2. Скрытые поля формы не являются authority.
3. Server Actions повторно проверяют владение.
4. Supabase service role key не попадает в браузер.
5. Raw initData не логируется.
6. Debug UI скрыт без `DEBUG_TELEGRAM_WEBAPP=true`.
7. Fallback-режим требует серверное окружение.
8. Fallback-продавец должен совпасть с fallback-магазином.

## Критерии приёмки

1. Владелец видит только один магазин, определённый на сервере.
2. Счётчики отчёта совпадают с правилами выручки уровня позиции.
3. Счётчик проверки показывает активные нерешённые позиции.
4. Редактирование сохраняется после перезагрузки.
5. Удаление сохраняется после перезагрузки.
6. Восстановление возвращает позицию в активное/удалённое состояние согласно предыдущему статусу.
7. Сброс дня не удаляет продажи физически.
8. Журнал записей показывает исходный голосовой контекст.
9. Статистика продавцов не включает cancelled/failed/deleted строки.
10. Ошибки auth/БД не маскируются как пустая успешная панель.
