# Дорожная карта

Дорожная карта описывает порядок развития проекта после текущей стабилизации. Актуальные системные правила продукта находятся в [../specs/global.md](../specs/global.md).

## Текущее состояние

Проект готовится к сдаче как Telegram-бот + Telegram WebApp для реального магазина.

Реализовано:

1. Голосовой конвейер сохраняет уверенные продажи как готовые.
2. Сомнительные позиции отправляются на проверку.
3. Telegram показывает только `✅ Подтвердить` и `❌ Отмена` для записи на проверке.
4. WebApp содержит вкладки `Отчёт`, `Проверка`, `Записи`, `Продавцы`.
5. WebApp подтверждает и отменяет записи через серверные действия.
6. Редактирование позиции сохраняется в Supabase.
7. Мягкое удаление исключает позицию из активной выручки.
8. Выручка считается по активным обработанным позициям.
9. Диагностика Telegram скрыта от обычного production-пользователя.

## Этап 0 - стабилизация перед сдачей

Цель: подтвердить, что текущий MVP работает в production-подобной среде.

Работы:

1. Проверить переменные окружения.
2. Применить миграции Supabase.
3. Настроить Telegram webhook.
4. Проверить `allowed_updates`.
5. Выполнить реальную голосовую продажу через Telegram.
6. Проверить подтверждение и отмену.
7. Проверить WebApp внутри Telegram.
8. Проверить журнал записей и аудио.
9. Проверить отсутствие секретов в клиентских ассетах и логах.

Release gate:

1. `npm.cmd run lint` прошёл.
2. `npm.cmd run test` прошёл.
3. `npm.cmd run build` прошёл.
4. `npm.cmd run web:build` прошёл.
5. Production smoke Telegram прошёл.
6. WebApp smoke в Telegram прошёл.
7. Нет критических ошибок auth, БД или webhook.

Риски:

1. Неверный webhook URL.
2. Webhook без `callback_query`.
3. Отсутствующие или неприменённые миграции.
4. Ошибки Telegram initData в реальном Mini App.
5. Недоступный Storage bucket для аудио.

## Этап 1 - первая эксплуатация

Цель: безопасно пройти первые реальные продажи и собрать обратную связь.

Работы:

1. Наблюдать логи Vercel после первых голосовых сообщений.
2. Проверить Supabase audit logs.
3. Сравнить ручной итог продаж с отчётом WebApp.
4. Собрать примеры фраз, которые попадают в проверку.
5. Зафиксировать частые ошибки продавцов.
6. Проверить удобство вкладки `Проверка` на телефоне.

Release gate:

1. Отчёт совпадает с ручной проверкой на контрольной выборке.
2. Сомнительные записи не попали в выручку без решения.
3. Отмена не оставляет выручку.
4. Редактирование не создаёт дублей.

Риски:

1. Продавцы диктуют фразы в неожиданном формате.
2. Слишком много позиций попадает в проверку.
3. Владельцу не хватает экспортируемого отчёта.

## Этап 2 - отчёты и экспорт

Цель: дать владельцу удобный способ выгрузить данные.

Планируемые функции:

1. Экспорт отчёта в CSV.
2. Экспорт отчёта в XLSX.
3. Фильтр по товару.
4. Фильтр по продавцу.
5. Улучшенное отображение итогов за период.

Release gate:

1. Экспорт считает выручку по тем же правилам, что WebApp.
2. Экспорт не включает позиции на проверке, отменённые и мягко удалённые строки.
3. Экспорт доступен только авторизованному пользователю текущего магазина.

Риски:

1. Расхождение между экспортом и экраном отчёта.
2. Утечка данных другого магазина.
3. Некорректная локаль дат и сумм.

## Этап 3 - управление продавцами

Цель: дать владельцу больше контроля над доступом продавцов.

Планируемые функции:

1. Просмотр активных и неактивных продавцов.
2. Деактивация продавца.
3. Создание или привязка продавца владельцем.
4. Проверка статистики по продавцу.
5. Более понятные ошибки для непривязанного продавца.

Release gate:

1. Деактивированный продавец не может сохранять продажи.
2. Владелец не может управлять продавцами чужого магазина.
3. Изменения доступа отражаются в Telegram и WebApp.

Риски:

1. Ошибка в owner/seller resolver.
2. Случайная потеря доступа продавца.
3. Неконсистентность между Telegram и WebApp.

## Этап 4 - аудит и восстановление

Цель: сделать исправления безопаснее и прозрачнее.

Планируемые функции:

1. Экран мягко удалённых позиций.
2. Восстановление позиции с пересчётом отчёта.
3. История действий подтверждения, отмены, редактирования и удаления.
4. Отдельные причины исключения позиции.

Release gate:

1. Восстановление возвращает правильный предыдущий статус.
2. Отчёт пересчитывается после восстановления.
3. Audit log failure не блокирует пользовательское действие.

Риски:

1. Восстановление ошибочно возвращает сомнительную позицию в выручку.
2. Пользователь не понимает разницу между отменой записи и удалением позиции.

## Этап 5 - качество распознавания

Цель: понять, где STT/LLM ошибаются, без переписывания конвейера.

Планируемые функции:

1. Аналитика доли `needs_review`.
2. Отчёт по причинам проверки.
3. Отчёт по сбоям STT.
4. Сбор частых неполных фраз.
5. Подсказки продавцам по диктовке.

Release gate:

1. Диагностика не раскрывает секреты.
2. Raw initData и токены не попадают в логи.
3. Аналитика не меняет правила сохранения продажи.

Риски:

1. Слишком подробные логи могут раскрыть чувствительные данные.
2. Попытка улучшить аналитику может незаметно изменить парсер.

## Вне рамок ближайших этапов

1. Замена STT-провайдера.
2. Замена LLM/parser-контракта.
3. Складской учёт.
4. Кассовая интеграция.
5. Платежи.
6. Полная desktop admin-панель.
7. Мульти-магазинный переключатель.
