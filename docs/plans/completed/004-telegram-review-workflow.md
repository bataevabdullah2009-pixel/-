# 004 - Сценарий проверки Telegram WebApp

Статус: завершённый исторический план.

Актуальное поведение проверки описано в [../../specs/global.md](../../specs/global.md), [../../specs/product/telegram-confirmation-flow.md](../../specs/product/telegram-confirmation-flow.md) и [../../features/manual-review.md](../../features/manual-review.md).

## Контекст

После появления голосового ввода стало нужно дать пользователю способ исправлять неполные или сомнительные позиции. На этом этапе проверка была ещё ранней и позже несколько раз уточнялась.

Основные проблемы:

1. Распознавание могло вернуть товар без цены.
2. Распознавание могло вернуть цену без понятного товара.
3. Пользователь должен был исправить позицию без прямого доступа к БД.
4. Ошибочную позицию нужно было убрать из активного отчёта без физического удаления.
5. UI должен был различать активные, проверяемые и исключённые позиции.

## Цель

Добавить пользовательский сценарий проверки:

1. Показать сомнительные позиции.
2. Позволить исправить товар, количество и цену.
3. Позволить исключить позицию.
4. Позволить восстановить мягко удалённую позицию.
5. Сохранить историю действий.

## Объём работ

В план входило:

1. UI для корректировки товаров.
2. Отдельные блоки активных, проверяемых и исключённых позиций.
3. Действие подтверждения для устаревших случаев проверки.
4. Мягкое удаление позиции.
5. Восстановление позиции.
6. Пересчёт отчёта после изменений.

Не входило:

1. Финальный UX вкладки `/review`.
2. Короткий callback-формат Telegram.
3. Подтверждение смешанной корзины на уровне отдельных позиций.
4. Полировка компактной карточки товара.

## Что было сделано

1. Добавлен интерфейс редактирования позиции.
2. Добавлена возможность исправить товар, количество и цену.
3. Добавлено мягкое удаление.
4. Добавлено восстановление для исключённых позиций.
5. Отчёт начал пересчитываться после пользовательских действий.
6. Сценарий проверки стал частью WebApp.

## Критерии закрытия

План считался закрытым, когда:

1. Сомнительная позиция была видна пользователю.
2. Пользователь мог исправить цену или товар.
3. Исправление сохранялось.
4. Удаление не удаляло строку физически.
5. Отчёт обновлялся после изменения.

## Что изменилось позже

Позже поведение было серьёзно уточнено:

1. Ручное сохранение валидной позиции сразу делает её готовой.
2. Новые уверенные голосовые позиции не требуют обязательной проверки.
3. Вкладка `/review` стала основным пользовательским экраном проверки.
4. Telegram-сообщение проверки содержит только две кнопки.
5. Кнопка `Открыть отчёт` удалена из сообщения проверки.
6. Callback-данные стали короткими: `confirm:<sale_id>` и `cancel:<sale_id>`.
7. Подтверждение смешанной корзины подтверждает валидные позиции и оставляет неполные.
8. Отмена переводит родительскую запись в `cancelled`.

## Текущее продуктовое правило

Сейчас проверка устроена так:

1. Сомнительная позиция показывается во вкладке `Проверка`.
2. Валидная исправленная позиция может войти в выручку.
3. Неполная соседняя позиция остаётся на проверке.
4. Отмена отменяет всю голосовую запись.
5. Удаление позиции выполняется через soft delete.
