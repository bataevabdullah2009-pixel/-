# 009 — продуктовая полировка позиции продажи WebApp

Статус: завершено 25 июня 2026.

## Цель

Исправить пользовательский поток сохранения/удаления товара и привести мобильные карточки отчёта к продуктовому состоянию без изменений голоса, STT, парсера и webhook.

## Причина

Форма редактирования была постоянно раскрыта, update выполнял полный redirect, pending/error-состояние отсутствовало, а удаление находилось в длинной текстовой ссылке под карточкой.

## Сделано

- Добавлена клиентская карточка `SaleItemCard`.
- Update/delete Server Actions возвращают структурированное состояние.
- Ошибка сохраняет введённые значения и остаётся внутри карточки.
- Кнопка сохранения показывает загрузку и блокируется.
- Update возвращает фактически сохранённую строку Supabase.
- Delete использует мягкое удаление и локальное подтверждение.
- Активный UI исключает и `deleted_at`, и устаревший `status = excluded`.
- Итоги обновляются через revalidation и `router.refresh()`.
- Фильтры периода уплотнены для мобильного экрана.
- Добавлено пустое состояние для продажи без активных товаров.
- Устранено Telegram CSS-variable hydration warning.

## Проверка

- Live-схема: `product_name`, `quantity`, `price`, `total`, `status`, `deleted_at`, `updated_at` существуют.
- Live WebApp: `5×50 → 10×50 → 10×100`, перезагрузка сохранила значения.
- Исходные `5×50` восстановлены после проверки.
- Мягкое удаление уменьшило итог; перезагрузка не вернула позицию; восстановление вернуло исходный итог.
- `npm run lint` — пройдено.
- `npm run test` — 8 файлов, 87 тестов пройдены.
- `npm run build` — bot/web/shared пройдены.

## Границы

Голосовой конвейер, STT, парсер, webhook, env-файлы и схема сохранения голосовых продаж не изменялись. Новая миграция не потребовалась.
