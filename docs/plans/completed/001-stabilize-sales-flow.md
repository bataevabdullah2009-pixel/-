# 001 - Стабилизация сценария продаж

Статус: завершённый исторический план.

Актуальное поведение продукта описано в [../../specs/00-global.md](../../specs/00-global.md). Этот документ фиксирует ранний этап стабилизации и может содержать решения, которые позже были уточнены последующими планами.

## Контекст

На раннем этапе продукт уже принимал голосовые продажи, но сценарий был недостаточно устойчивым для проверки:

1. Нужно было защитить отчёт от чужих данных магазина.
2. Нужно было стабилизировать расчёт выручки.
3. Нужно было сохранить сомнительные распознавания вместо потери записи.
4. Нужно было добавить мягкое удаление позиций.
5. Нужно было покрыть критические сценарии тестами.

## Цель

Сделать базовый поток продаж предсказуемым:

1. Продавец отправляет голос.
2. Система сохраняет запись.
3. Уверенные данные попадают в отчёт.
4. Сомнительные данные не ломают выручку.
5. Ошибочные позиции можно исключить без физического удаления.

## Объём работ

В план входило:

1. Базовая изоляция магазина.
2. Расчёт отчёта по текущему магазину.
3. Поля мягкого удаления для позиций.
4. Проверка доступа продавца.
5. Резервный парсер для recoverable ошибок.
6. Регрессионные тесты для продаж и отчёта.

Не входило:

1. Полная модель владельца.
2. Расширенная авторизация Telegram WebApp.
3. Production webhook hardening.
4. Полная полировка UX проверки.

## Что было сделано

1. Уточнена граница магазина для чтения отчёта.
2. Добавлена защита от попадания чужих продаж в текущий отчёт.
3. Добавлены поля и правила мягкого удаления `sale_items`.
4. Сценарий продавца стал проверять активную привязку.
5. Резервный парсер стал сохранять данные для проверки вместо тихого сбоя.
6. Добавлены регрессионные тесты для ключевых переходов.

## Критерии закрытия

План считался закрытым, когда:

1. Отчёт не смешивал магазины.
2. Уверенная продажа сохранялась.
3. Сомнительная продажа не терялась.
4. Исключённая позиция не входила в активную выручку.
5. Тесты покрывали базовый сценарий сохранения и отчёта.

## Что изменилось позже

Последующие планы уточнили ранние решения:

1. Уверенные голосовые позиции теперь могут сразу сохраняться как `processed`.
2. WebApp получил явный fallback-режим.
3. Подтверждение стало работать на уровне активных `sale_items`.
4. Смешанная корзина больше не блокируется одной неполной позицией.
5. Отмена голосовой записи использует `cancelled`.
