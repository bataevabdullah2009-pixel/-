# 015 - Парсер sale_items, подтверждение и выручка

Дата: 2026-07-03

## Цель

Исправить конвейер `voice transcript -> parser -> sale_items -> confirm -> revenue`, где фраза с несколькими позициями сохранялась одной неполной строкой.

## Причина

Детерминированный fallback делил текст по запятой слишком рано. Фраза `Сникерс, 3 штуки по 200 рублей` превращалась в отдельные сегменты `Сникерс` и `3 штуки по 200 рублей`, из-за чего полная позиция не создавалась. Когда LLM возвращал одну длинную позицию, сохранение получало одну позицию с полным текстом, `quantity = 1`, `price = null`, `total = null`.

## Сделано

- Резервный парсер теперь ищет полные доказательства quantity/price по всей фразе и не ломает запятую между названием и количеством.
- `Сникерс, 3 штуки по 200 рублей. Буханка хлеба, 5 штук по 50 рублей.` даёт две `sale_items` и сумму `850`.
- Поддержаны `Сникерс 5 по 100`, `Пицца 1 штука 500 рублей`, `Кола 2 бутылки по 150 рублей`, `Хлеб 3 штуки по 50, шоколад 2 штуки по 100`.
- Неполные фрагменты вроде `Корзина продуктов` сохраняются отдельными строками `needs_review`.
- Невалидный JSON от LLM/parser fallback тоже использует детерминированное извлечение позиций.
- Логи подтверждения теперь включают sale id, found count, valid count и invalid reasons.
- Сообщение успешного подтверждения: `✅ Подтверждено: N позиций, сумма X ₽`.
- Заменено 2026-07-03: ручное сохранение в WebApp сохраняет поля позиции, пересчитывает `total`, ставит позицию `processed`, и эта позиция может войти в выручку, пока родительская `needs_review` остаётся из-за неполных соседних позиций.

## Проверка

- `npm.cmd run test`: 8 файлов, 109 тестов.
- `npm.cmd run lint`.
- `npm.cmd run build`.
- `npm.cmd run web:build`.
- Read-only запрос Supabase показал существующие склеенные строки до исправления в свежих продажах.
- Supabase RPC smoke вставил временную продажу с двумя строками:
  - `Сникерс`, `3 шт`, `200`, `600`;
  - `Буханка хлеба`, `5 шт`, `50`, `250`.
- Сумма smoke-продажи была `850`; созданные `sales`, `sale_items`, `voice_records` были затем удалены, а проверка очистки вернула ноль строк.
