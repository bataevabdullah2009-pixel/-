# Продукт

Голосовой журнал продаж решает одну задачу: быстро фиксировать продажи без таблиц и ручного ввода во время смены. Продавец говорит продажу в Telegram, система сохраняет структурированные позиции, а владелец видит отчёт и исправляет спорные строки в WebApp.

## Пользовательский сценарий

1. Продавец говорит: `Сникерс 4 штуки по 100 рублей`.
2. Бот получает голосовое сообщение.
3. Бот проверяет, что пользователь Telegram привязан к активному продавцу.
4. Бот скачивает аудио.
5. Бот готовит аудио для STT.
6. STT возвращает русскую расшифровку.
7. LLM очищает текст.
8. Парсер выделяет товары, количество, единицу, цену и сумму.
9. Детерминированный слой доказательств сверяет вывод парсера с расшифровкой.
10. Supabase сохраняет голосовую запись, продажу и позиции продажи.
11. Уверенная позиция сразу становится `processed`.
12. Неполная позиция становится `needs_review`.
13. Владелец открывает WebApp.
14. Владелец видит отчёт, проверку, записи и продавцов.
15. Владелец исправляет товар карандашом или исключает корзиной.

## Что считается готовой продажей

Позиция готова, если:

1. Есть осмысленный товар.
2. Есть количество или вес.
3. Есть цена или сумма, из которой можно восстановить цену.
4. Сумма валидна.
5. Уверенность достаточно высокая.

Готовая позиция:

1. Получает статус `processed`.
2. Входит в выручку.
3. Показывается в отчёте.
4. Попадает в топ товаров.
5. Учитывается в статистике продавца.

## Что считается сомнительной продажей

Сомнительная позиция появляется, если:

1. Не выделен отдельный товар.
2. Нет количества.
3. Нет веса.
4. Нет цены.
5. Нет суммы.
6. Уверенность низкая.
7. Резервный парсер не может подтвердить доказательства.

Сомнительная позиция:

1. Получает `needs_review`.
2. Не входит в выручку.
3. Появляется на вкладке `Проверка`.
4. Может быть подтверждена после исправления.
5. Может быть отменена вместе с родительской продажей.

## Смешанная корзина

Одна голосовая запись может содержать и готовые, и неполные позиции.

Пример:

```text
Хлеб 3 штуки по 50 рублей. Корзина продуктов.
```

Ожидаемое поведение:

1. `Хлеб` сохраняется как полноценная позиция.
2. `Корзина продуктов` сохраняется как позиция на проверку.
3. Родительская продажа может остаться `needs_review`.
4. Активная позиция `processed` может войти в выручку.
5. Неполная позиция остаётся в `Проверке`.
6. Подтверждение подтверждает валидные позиции и оставляет неполные.
7. Отмена отменяет всю голосовую запись и мягко удаляет активные позиции.

## WebApp

Владелец видит:

1. `Отчёт`.
2. `Проверка`.
3. `Записи`.
4. `Продавцы`.

`Отчёт` показывает:

1. Выручку.
2. Количество товаров.
3. Количество записей.
4. Количество позиций на проверку.
5. Топ товаров.
6. Продажи по дням.
7. Последние продажи.
8. Исключённые товары.

`Проверка` показывает активные позиции на проверке и действия:

1. Подтвердить.
2. Отмена.
3. Подтвердить всё.
4. Редактировать позицию.
5. Мягко удалить позицию.

`Записи` показывает исходный контекст голосовой продажи.

`Продавцы` показывает количество записей, активность и выручку за период.

## Пользовательские статусы

Пользователь видит:

1. `Готово`.
2. `Нужно проверить`.
3. `Исключено`.

Внутренние значения enum пользователю не показываются.

Отображение:

1. `processed` -> `Готово`.
2. `needs_review` -> `Нужно проверить`.
3. `needs_price` -> `Нужно проверить`.
4. `pending` -> `Нужно проверить`.
5. `failed` -> `Нужно проверить`.
6. `cancelled` -> `Исключено`.
7. `excluded` -> `Исключено`.

## Редактирование

Карандаш раскрывает:

1. Товар.
2. Количество.
3. Единицу.
4. Цену.

Сохранение:

1. Обновляет строку Supabase.
2. Пересчитывает сумму позиции.
3. Ставит позицию `processed`.
4. Пересчитывает родительскую продажу.
5. Обновляет отчёт без ручной перезагрузки.

Если родительская продажа остаётся `needs_review`, исправленная позиция всё равно может войти в выручку.

## Удаление

Корзина исключает товар из активного отчёта:

1. Физического удаления нет.
2. Позиция получает `status = excluded`.
3. Позиция получает `deleted_at`.
4. Предыдущий статус сохраняется.
5. Позицию можно восстановить.
6. Родительская продажа остаётся в журнале.

## Доступ

1. WebApp определяет магазин на сервере.
2. Client `shop_id` игнорируется.
3. Telegram initData проверяется HMAC.
4. Резервный режим включается только флагом окружения.
5. Service role key остаётся на сервере.

## Что продукт не делает сейчас

1. Не ведёт складской учёт.
2. Не списывает остатки.
3. Не строит бухгалтерскую отчётность.
4. Не даёт переключатель нескольких магазинов.
5. Не даёт публичные клиентские бизнес-записи.
