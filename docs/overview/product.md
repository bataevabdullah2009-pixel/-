# Продукт

Voice Sales Log решает одну задачу: быстро фиксировать продажи без таблиц и ручного ввода во время смены. Продавец говорит продажу в Telegram, система сохраняет структурированные позиции, а владелец видит отчёт и исправляет спорные строки в WebApp.

## Пользовательский сценарий

1. Продавец говорит: `Сникерс 4 штуки по 100 рублей`.
2. Бот получает voice message.
3. Бот проверяет, что Telegram user привязан к active seller.
4. Бот скачивает аудио.
5. Бот готовит аудио для STT.
6. STT возвращает русский transcript.
7. LLM очищает текст.
8. Parser выделяет товары, количество, единицу, цену и total.
9. Deterministic evidence layer сверяет parser output с transcript.
10. Supabase сохраняет voice record, sale и sale items.
11. Уверенная позиция сразу становится `processed`.
12. Неполная позиция становится `needs_review`.
13. Владелец открывает WebApp.
14. Владелец видит отчёт, проверку, записи и продавцов.
15. Владелец исправляет товар карандашом или исключает корзиной.

## Что считается готовой продажей

Позиция готова, если:

1. Есть осмысленный товар.
2. Есть количество или вес.
3. Есть цена или total, из которого можно восстановить цену.
4. Total валиден.
5. Confidence достаточно высокий.

Готовая позиция:

1. Получает статус `processed`.
2. Входит в выручку.
3. Показывается в отчёте.
4. Попадает в топ товаров.
5. Учитывается в статистике продавца.

## Что считается сомнительной продажей

Сомнительная позиция появляется, если:

1. Не выделен отдельный товар.
2. Нет количества.
3. Нет веса.
4. Нет цены.
5. Нет total.
6. Confidence низкий.
7. Parser fallback не может подтвердить evidence.

Сомнительная позиция:

1. Получает `needs_review`.
2. Не входит в выручку.
3. Появляется на вкладке `Проверка`.
4. Может быть подтверждена после исправления.
5. Может быть отменена вместе с parent sale.

## Mixed cart

Одна voice-запись может содержать и готовые, и неполные позиции.

Пример:

```text
Хлеб 3 штуки по 50 рублей. Корзина продуктов.
```

Ожидаемое поведение:

1. `Хлеб` сохраняется как полноценный item.
2. `Корзина продуктов` сохраняется как review item.
3. Parent sale может остаться `needs_review`.
4. Active `processed` item может войти в выручку.
5. Неполный item остаётся в `Проверке`.
6. Confirm подтверждает валидные items и оставляет неполные.
7. Cancel отменяет всю voice-запись и soft-delete active items.

## WebApp

Владелец видит:

1. `Отчёт`.
2. `Проверка`.
3. `Записи`.
4. `Продавцы`.

`Отчёт` показывает:

1. Выручку.
2. Количество товаров.
3. Количество записей.
4. Количество позиций на проверку.
5. Топ товаров.
6. Продажи по дням.
7. Последние продажи.
8. Исключённые товары.

`Проверка` показывает active review items и actions:

1. Подтвердить.
2. Отмена.
3. Подтвердить всё.
4. Edit item.
5. Soft delete item.

`Записи` показывает исходный контекст voice sale.

`Продавцы` показывает records count, activity и revenue за период.

## Пользовательские статусы

Пользователь видит:

1. `Готово`.
2. `Нужно проверить`.
3. `Исключено`.

Internal enum значения пользователю не показываются.

Mapping:

1. `processed` -> `Готово`.
2. `needs_review` -> `Нужно проверить`.
3. `needs_price` -> `Нужно проверить`.
4. `pending` -> `Нужно проверить`.
5. `failed` -> `Нужно проверить`.
6. `cancelled` -> `Исключено`.
7. `excluded` -> `Исключено`.

## Редактирование

Карандаш раскрывает:

1. Товар.
2. Количество.
3. Единицу.
4. Цену.

Сохранение:

1. Обновляет Supabase row.
2. Пересчитывает item total.
3. Ставит item `processed`.
4. Пересчитывает parent sale.
5. Обновляет отчёт без ручного reload.

Если parent sale остаётся `needs_review`, исправленная позиция всё равно может войти в выручку.

## Удаление

Корзина исключает товар из активного отчёта:

1. Физического удаления нет.
2. Item получает `status = excluded`.
3. Item получает `deleted_at`.
4. Previous status сохраняется.
5. Item можно восстановить.
6. Parent sale остаётся в журнале.

## Доступ

1. WebApp определяет shop на сервере.
2. Client `shop_id` игнорируется.
3. Telegram initData проверяется HMAC.
4. Fallback mode включается только env флагом.
5. Service role key остаётся server-side.

## Что продукт не делает сейчас

1. Не ведёт складской учёт.
2. Не списывает остатки.
3. Не строит бухгалтерскую отчётность.
4. Не даёт multi-shop switcher.
5. Не даёт публичные client-side business writes.
