# Продуктовый документ

`Голосовой журнал продаж` - Telegram-бот и Telegram WebApp для магазина, где продавцы фиксируют продажи голосом, а владелец видит понятный отчёт по выручке и может быстро исправлять спорные позиции.

Этот документ описывает продукт простым языком: для кого он, какую проблему решает, какие сценарии поддерживает и что считается корректным поведением с точки зрения пользователя. Полный системный контракт со статусами, правилами выручки и тестовыми сценариями находится в [глобальной спецификации](../specs/00-global.md).

## 1. Проблема

В небольшом магазине продавцу неудобно вести продажи вручную во время смены:

1. Некогда заполнять таблицу после каждой продажи.
2. Легко забыть часть товаров или цену.
3. В конце дня сложно быстро понять выручку.
4. Владельцу нужно видеть продажи без постоянных уточнений у продавца.
5. Ошибки распознавания нельзя автоматически включать в выручку.

Продукт решает это через голосовой ввод: продавец говорит продажу в Telegram, а система превращает речь в структурированные строки отчёта.

## 2. Ценность

Для продавца:

1. Не нужно открывать таблицу или форму.
2. Достаточно отправить голосовое сообщение в Telegram.
3. Бот сразу отвечает, что запись сохранена или требует проверки.
4. Сомнительные записи можно подтвердить прямо под сообщением.

Для владельца:

1. Видна выручка за выбранный период.
2. Видно, какие товары проданы.
3. Видны записи, которые нужно проверить.
4. Можно исправить товар, количество или цену в WebApp.
5. Ошибочные позиции можно исключить из отчёта без потери истории.

## 3. Пользователи

### Продавец

Продавец работает в Telegram:

1. Запускает бота.
2. Отправляет голосовые сообщения с продажами.
3. Получает ответ об успешном сохранении или проверке.
4. При необходимости нажимает `✅ Подтвердить` или `❌ Отмена`.
5. Может открыть WebApp, если ему разрешён доступ.

### Владелец

Владелец работает в WebApp:

1. Смотрит сводку магазина.
2. Проверяет сомнительные позиции.
3. Исправляет товары и цены.
4. Исключает ошибочные позиции.
5. Смотрит журнал записей и продавцов.

### Система

Система автоматически:

1. Распознаёт голос.
2. Выделяет товары, количество, единицы и цены.
3. Делит склеенные фразы на отдельные позиции.
4. Отправляет сомнительные данные на проверку.
5. Считает выручку только из безопасных обработанных позиций.

## 4. Основной пользовательский сценарий

Пример:

```text
Сникерс четыре штуки по сто рублей, хлеб два по пятьдесят.
```

Ожидаемый поток:

1. Продавец отправляет голосовое сообщение в Telegram.
2. Бот принимает сообщение и определяет продавца.
3. Система получает расшифровку голоса.
4. Парсер выделяет две позиции: `Сникерс` и `хлеб`.
5. Каждая позиция получает количество, цену и сумму.
6. Supabase сохраняет голосовую запись, продажу и позиции.
7. Бот отвечает, что запись сохранена.
8. В WebApp обновляется отчёт.
9. Выручка увеличивается на сумму обработанных позиций.

## 5. Сценарий уверенной продажи

Продажа считается уверенной, если система понимает товар, количество или вес, цену и сумму.

Пример:

```text
Буханка хлеба пять штук по сто рублей.
```

Результат:

1. Запись сохраняется без ручной проверки.
2. Позиция получает состояние `Готово`.
3. Продажа появляется в `Отчёте`.
4. Товар участвует в топе товаров.
5. Продавец получает обычный ответ об успешном сохранении.

Пользователь не должен видеть технические статусы вроде `processed`; в интерфейсе это отображается как готовая запись или готовая позиция.

## 6. Сценарий сомнительной продажи

Продажа или позиция считается сомнительной, если системе не хватает данных.

Примеры:

```text
Корзина продуктов.
```

```text
Сникерс несколько штук.
```

```text
Хлеб, молоко, ещё там что-то.
```

Результат:

1. Запись сохраняется.
2. Сомнительная позиция не входит в выручку.
3. Бот показывает предупреждение.
4. Под сообщением есть только две кнопки: `✅ Подтвердить` и `❌ Отмена`.
5. Позиция появляется во вкладке `Проверка`.
6. Владелец может исправить её в WebApp.

Кнопка `Открыть отчёт` не показывается в сообщении проверки, чтобы пользователь не путал решение записи с переходом в отчёт.

## 7. Смешанная корзина

Одна голосовая запись может содержать и понятные, и непонятные позиции.

Пример:

```text
Хлеб три штуки по пятьдесят рублей. Корзина продуктов.
```

Ожидаемое поведение:

1. `Хлеб` сохраняется как полноценная позиция.
2. `Корзина продуктов` сохраняется как позиция на проверку.
3. Полная позиция может войти в выручку.
4. Неполная позиция остаётся во вкладке `Проверка`.
5. Подтверждение не должно блокировать всю корзину из-за одной неполной позиции.
6. Отмена отменяет всю голосовую запись и убирает её активные позиции из выручки.

Это важное продуктовое правило: выручка считается на уровне позиции, а не только по общему статусу голосовой записи.

## 8. Подтверждение

Подтверждение нужно, когда пользователь согласен с полными позициями в сомнительной записи.

Что происходит:

1. Система проверяет активные позиции записи.
2. Полные позиции становятся готовыми.
3. Неполные позиции остаются на проверке.
4. Сумма продажи пересчитывается.
5. Пользователь получает сообщение вида `✅ Подтверждено: N позиций, сумма X ₽`.

Если в записи нет ни одной полной позиции, подтверждение не выполняется. Пользователь видит понятную ошибку: `Не удалось подтвердить: нет ни одной полной позиции.`

## 9. Отмена

Отмена нужна, когда пользователь отклоняет сомнительную голосовую запись.

Что происходит:

1. Запись получает состояние отмены.
2. Активные позиции мягко удаляются.
3. Выручка по этой записи становится нулевой.
4. История остаётся в системе для аудита.
5. Повторная отмена не ломает данные.

Отмена не удаляет продажу физически.

## 10. WebApp

WebApp - рабочая панель для просмотра и исправления продаж.

### Отчёт

Показывает:

1. Выручку за период.
2. Количество проданных товаров.
3. Количество голосовых записей.
4. Количество позиций на проверку.
5. Топ товаров.
6. Продажи за период.
7. Динамику по дням.

Периоды:

1. Сегодня.
2. Вчера.
3. Неделя.
4. Месяц.
5. Пользовательская дата.

### Проверка

Показывает только активные сомнительные позиции.

Пользователь может:

1. Подтвердить запись.
2. Отменить запись.
3. Подтвердить всё доступное.
4. Отредактировать позицию.
5. Исключить позицию из отчёта.

### Записи

Показывает журнал голосовых продаж:

1. Дату.
2. Продавца.
3. Распознанный текст.
4. Статус.
5. Сумму.
6. Список товаров.
7. Аудио при наличии доступной ссылки.

### Продавцы

Показывает:

1. Список продавцов.
2. Активность за выбранный период.
3. Количество записей.
4. Выручку по каждому продавцу.

## 11. Редактирование позиции

Редактирование открывается через карандаш.

Поля:

1. Товар.
2. Количество.
3. Единица.
4. Цена.

После сохранения:

1. Данные сохраняются в Supabase.
2. Сумма позиции пересчитывается.
3. Позиция становится готовой, если данные полные.
4. Итоги отчёта обновляются.

Единицы:

1. `шт`.
2. `кг`.
3. `г`.

Для граммов сумма считается как доля килограмма от цены за кг.

## 12. Удаление позиции

Удаление открывается через корзину.

Правила:

1. Пользователь видит подтверждение удаления.
2. Позиция исключается из активного отчёта.
3. Физического удаления нет.
4. Предыдущий статус сохраняется.
5. После перезагрузки удалённая позиция не возвращается в активный отчёт.

## 13. Пользовательские состояния

Пользователь видит понятные состояния, а не внутренние enum.

1. `Готово` - позиция обработана и может участвовать в выручке.
2. `Нужно проверить` - позиция требует решения или исправления.
3. `Исключено` - позиция отменена или мягко удалена.

Подробная таблица соответствий описана в [глобальной спецификации](../specs/00-global.md#8-статусы-записей) и [жизненном цикле статусов](../specs/41-data-status-lifecycle.md).

## 14. Правила выручки для пользователя

В отчёт попадает только то, что можно безопасно считать продажей.

Входит:

1. Активная готовая позиция.
2. Позиция с валидной суммой.
3. Позиция из продажи, которая не отменена и не завершилась ошибкой.

Не входит:

1. Позиция на проверке.
2. Позиция без цены или суммы.
3. Отменённая запись.
4. Ошибочная запись.
5. Мягко удалённая позиция.

Техническое правило расчёта описано в [глобальной спецификации](../specs/00-global.md#17-правила-выручки).

## 15. Доступ и безопасность

Продукт должен защищать данные магазина:

1. WebApp определяет магазин на сервере.
2. Клиент не выбирает магазин сам.
3. Telegram initData проверяется сервером.
4. Service role key не попадает в браузер.
5. Ошибка доступа не должна отображаться как пустой отчёт.
6. Диагностика Telegram скрыта в production без специального флага.

Подробности: [роли и доступ](../specs/11-product-roles-and-access.md), [авторизация и изоляция магазина](../specs/24-technical-auth-and-shop-isolation.md).

## 16. Границы MVP

В текущий продукт входит:

1. Голосовой ввод продаж в Telegram.
2. Распознавание и разбор русской речи.
3. Сохранение продаж и позиций в Supabase.
4. Проверка сомнительных позиций.
5. Подтверждение и отмена в Telegram.
6. Проверка и редактирование в WebApp.
7. Отчёт по выручке.
8. Журнал записей.
9. Статистика продавцов.
10. Мягкое удаление позиций.

Не входит:

1. Складской учёт.
2. Бухгалтерская отчётность.
3. Платежи.
4. Кассовая интеграция.
5. Полноценная desktop admin-панель.
6. Автоматическое списание остатков.

## 17. Критерии готовности продукта

Продукт готов к проверке, если:

1. Уверенная голосовая продажа появляется в отчёте.
2. Сомнительная продажа появляется в проверке.
3. Telegram показывает только две кнопки решения для сомнительной записи.
4. Подтверждение добавляет в выручку только полные позиции.
5. Отмена исключает запись из выручки.
6. Редактирование позиции сохраняется после перезагрузки.
7. Удаление позиции сохраняется после перезагрузки.
8. WebApp показывает четыре вкладки.
9. Отчёт не смешивает данные разных магазинов.
10. Ошибки доступа и БД видны пользователю как ошибки.
