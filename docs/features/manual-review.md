# Ручная проверка

Ручная проверка нужна для записей, где система не уверена в товаре, количестве, весе, цене, сумме или уверенности парсера. Проверка сохраняет данные, но не включает сомнительные позиции в выручку до исправления или подтверждения.

## Где выполняется проверка

1. В Telegram под сообщением сомнительной голосовой записи.
2. В WebApp на вкладке `Проверка`.
3. Через редактирование WebApp внутри карточки товара.
4. Через массовое действие `Подтвердить всё` на `/review`.

Callback Telegram и действия проверки WebApp используют одинаковые правила подтверждения/отмены на уровне позиции, но разные источники авторизации. Callback Telegram определяет продавца по `ctx.from.id`, действия WebApp определяют магазин через сессию Telegram WebApp или серверный резервный режим.

## Сообщение Telegram

Под сообщением бота есть только:

1. `✅ Подтвердить`.
2. `❌ Отмена`.

В сообщении проверки нет кнопки `Открыть отчёт`. Открытие отчёта остаётся в `/start`, reply keyboard и menu button.

Данные callback:

```text
confirm:<sale_id>
cancel:<sale_id>
```

Устаревший формат принимается для старых сообщений:

```text
voice_sale_review:confirm:<sale_id>
voice_sale_review:cancel:<sale_id>
```

## Проверка WebApp

`/review` показывает только активные позиции проверки:

1. `deleted_at is null`.
2. `status` не `processed`.
3. `status` не `excluded`.
4. Родительская продажа входит в область текущего магазина.
5. Родительская продажа не `cancelled` и не `failed`.

Экран показывает:

1. Название магазина.
2. Количество позиций.
3. Количество родительских продаж.
4. Потенциальную сумму по позициям проверки, если сумма уже есть.
5. DateFilter.
6. Карточки позиций.
7. Действие `Подтвердить`.
8. Действие `Отмена`.
9. Массовое действие `Подтвердить всё`.

## Карточка позиции проверки

Карточка показывает:

1. Название товара или `Без названия`.
2. Количество и единицу.
3. Цену или сообщение `цена не указана`.
4. Сумму или `Не входит в выручку`.
5. Бейдж `Нужно проверить`.
6. Причины проверки.
7. Карандаш для редактирования.
8. Корзину для мягкого удаления.

Причины проверки:

1. Нет отдельного товара.
2. Нет количества или веса.
3. Нет цены.
4. Нет суммы.
5. Низкая уверенность.

## Редактирование во время проверки

Карандаш открывает поля:

1. `Товар`.
2. `Количество`.
3. `Единица`.
4. `Цена, ₽`.

`Сохранить`:

1. Валидирует id позиции.
2. Валидирует название товара.
3. Валидирует quantity `> 0`.
4. Валидирует price `> 0`.
5. Проверяет позицию -> продажу -> магазин на сервере.
6. Обновляет строку Supabase.
7. Пересчитывает `total`.
8. Ставит позицию `processed`.
9. Ставит уверенность `1`.
10. Пересчитывает родительскую продажу.
11. Делает revalidate отчёта, проверки, записей и продавцов.

Если родительская продажа остаётся `needs_review`, это не мешает исправленной строке позиции войти в выручку. Другие неполные активные позиции той же продажи остаются на вкладке `Проверка`.

`Отмена` в форме редактирования сбрасывает несохранённый ввод и закрывает форму. Ошибка сохранения показывает сообщение внутри карточки и не очищает поля.

## Подтверждение

Подтверждение работает на уровне активных `sale_items`, а не только на уровне родительской продажи.

Алгоритм:

1. Найти продажу в текущем магазине.
2. Если продажа уже `processed`, вернуть `✅ Уже подтверждено`.
3. Если продажа уже `cancelled`, вернуть успех без изменений.
4. Если продажа `failed`, вернуть ошибку.
5. Загрузить активные позиции с `deleted_at is null`.
6. Исключить `status = excluded`.
7. Проверить каждую позицию отдельно.
8. Подтверждаемая позиция должна иметь осмысленный товар.
9. Подтверждаемая позиция должна иметь положительное количество или вес.
10. Подтверждаемая позиция должна иметь цену или сумму, из которой можно вывести цену.
11. Если нет ни одной подтверждаемой позиции, вернуть `Не удалось подтвердить: нет ни одной полной позиции.`
12. Все подтверждаемые позиции обновить до `processed`.
13. Уверенность подтверждаемых позиций поставить `1`.
14. Пересчитать сумму.
15. Если неполных активных позиций больше нет, родительская продажа/голосовая запись становятся `processed`.
16. Если неполные активные позиции остались, родительская продажа/голосовая запись остаются `needs_review`.
17. `sales.total_amount` становится суммой активных обработанных позиций.

Смешанная корзина подтверждается частично: хорошая позиция входит в выручку, плохая остаётся на проверке.

## Отмена

Отмена означает отмену всей голосовой записи.

Алгоритм:

1. Найти продажу в текущем магазине.
2. Если продажа уже `cancelled`, вернуть успех без изменений.
3. Если продажа уже `processed`, вернуть успех без изменений и не менять выручку.
4. Если продажа `failed`, вернуть ошибку в сценарии WebApp.
5. Загрузить активные позиции.
6. Для каждой активной позиции выполнить мягкое удаление.
7. Родительскую продажу перевести в `cancelled`.
8. Родительский `total_amount` поставить `0`.
9. Голосовую запись перевести в `cancelled`.
10. Сделать revalidate затронутых маршрутов.

После успешной отмены никакие активные позиции этой продажи не входят в выручку, включая те, которые до отмены были `processed`.

## Soft delete товара

Удаление отдельного товара через корзину не равно отмене всей голосовой записи.

Патч:

```text
status = excluded
deleted_at = now()
deleted_reason = excluded_by_owner
deleted_previous_status = previous status
updated_at = now()
```

Такая позиция:

1. Не входит в активный отчёт.
2. Не входит в список проверки.
3. Не входит в выручку.
4. Может быть восстановлен.
5. Не удаляет parent sale.

## Restore

Восстановление:

1. Доступно для мягко удалённой позиции.
2. Проверяет позицию -> продажу -> магазин.
3. Возвращает `deleted_previous_status`.
4. Если предыдущий статус неизвестен, возвращает `needs_review`.
5. Очищает metadata удаления.
6. Пересчитывает родительскую продажу.
7. Пишет audit log best-effort.

## Критерии приёмки

1. Сообщение проверки содержит только две кнопки.
2. Данные callback короткие.
3. Устаревший формат callback не ломает старые сообщения.
4. WebApp `/review` не показывает обработанные активные позиции.
5. WebApp `/review` не показывает удалённые/исключённые позиции.
6. Подтверждение смешанной корзины не блокируется одной неполной позицией.
7. Подтверждение без полной позиции не меняет данные.
8. Отмена всей продажи мягко удаляет активные позиции.
9. Редактирование позиции проверки может сделать конкретную позицию готовой к выручке.
10. Родительский `needs_review` сохраняется, пока есть нерешённые соседние позиции.
