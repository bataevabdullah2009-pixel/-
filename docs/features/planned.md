# Запланированные функции

Этот документ описывает продуктовый backlog после текущего MVP. Он не заменяет [глобальную спецификацию](../specs/global.md), а фиксирует, что делать дальше, что проверять перед сдачей и что не входит в ближайший релиз.

## Принципы планирования

1. Сначала защищать корректность выручки.
2. Не менять работающий голосовой конвейер без отдельной причины.
3. Не добавлять функции, которые требуют изменения схемы Supabase, без отдельной миграционной задачи.
4. Не расширять WebApp за счёт безопасности: все бизнес-данные остаются ограничены текущим магазином.
5. Любая новая функция должна иметь критерии готовности и сценарий проверки.

## Приоритеты

### P0 - обязательно перед сдачей

1. Production smoke с реальным Telegram-ботом.
2. Проверка callback-доставки `callback_query` после настройки webhook.
3. Проверка WebApp в реальном Telegram Mini App.
4. Проверка, что аудио из Supabase Storage доступно в журнале записей.
5. Проверка, что service role key не попадает в клиентские ассеты.
6. Проверка логов Vercel/Supabase после тестовой голосовой продажи.
7. Проверка сценариев уверенной, сомнительной, смешанной и отменённой продажи.

Критерии готовности P0:

1. Реальная уверенная продажа попадает в `Отчёт`.
2. Реальная сомнительная продажа попадает в `Проверку`.
3. `✅ Подтвердить` и `❌ Отмена` работают из Telegram.
4. WebApp открывается из Telegram без fallback.
5. Ошибки доступа не отображаются как пустые данные.
6. Все обязательные команды качества проходят или явно зафиксированы причины сбоя.

### P1 - сразу после сдачи

1. Экспорт отчёта в CSV.
2. Экспорт отчёта в XLSX.
3. Фильтр по товару в журнале записей.
4. Улучшенный экран восстановления мягко удалённых позиций.
5. Расширенная страница управления продавцами для владельца.
6. Более подробные пользовательские сообщения для ошибок WebApp.

Критерии готовности P1:

1. Экспорт совпадает с текущими правилами выручки.
2. Фильтры не обходят изоляцию магазина.
3. Восстановление soft-delete пересчитывает отчёт.
4. Управление продавцами не даёт доступ к чужому магазину.

### P2 - улучшения после стабилизации

1. Аналитика уверенности парсера.
2. Отдельный отчёт по сбоям STT/LLM.
3. Расширенный поиск по журналу записей.
4. Улучшенные подсказки для продавца по формату голосовой продажи.
5. Уведомления владельцу о большом количестве позиций на проверке.
6. Отдельный экран аудита действий подтверждения, отмены и удаления.

Критерии готовности P2:

1. Аналитика не раскрывает секреты и raw initData.
2. Новые отчёты используют только server-side доступ.
3. События аудита не блокируют основные пользовательские действия.

## Перед сдачей

Нужно выполнить:

1. Проверить production env.
2. Применить Supabase migrations.
3. Настроить Telegram webhook.
4. Проверить `allowed_updates` с `message` и `callback_query`.
5. Отправить тестовую уверенную голосовую продажу.
6. Отправить тестовую сомнительную голосовую продажу.
7. Нажать `✅ Подтвердить` в Telegram.
8. Нажать `❌ Отмена` в Telegram.
9. Проверить WebApp: `Отчёт`, `Проверка`, `Записи`, `Продавцы`.
10. Проверить редактирование позиции.
11. Проверить мягкое удаление позиции.
12. Проверить аудио в журнале записей.
13. Проверить логи Vercel и Supabase без секретов.

## После сдачи

Разрешено планировать:

1. Улучшение отчётов.
2. Экспорт данных.
3. Удобство работы владельца.
4. Восстановление мягко удалённых позиций.
5. Диагностику качества распознавания.
6. Административные функции для продавцов.

Нельзя считать "после сдачи" поводом для:

1. Переписывания STT.
2. Переписывания парсера.
3. Переписывания подтверждения и отмены.
4. Замены Supabase persistence.
5. Большого UI-редизайна без отдельного продуктового решения.

## Не входит в текущую область

1. Складской учёт.
2. Списание остатков.
3. Бухгалтерская отчётность.
4. Интеграция с кассой.
5. Платежи.
6. Полная desktop admin-панель.
7. Мульти-магазинный переключатель в UI.
8. Публичные клиентские записи в бизнес-таблицы.
9. Физическое удаление продаж и позиций как пользовательский сценарий.

## Как добавлять новую функцию

Перед добавлением функции нужно описать:

1. Пользовательскую проблему.
2. Роль пользователя.
3. Основной сценарий.
4. Изменения в WebApp или Telegram.
5. Влияние на выручку.
6. Влияние на Supabase schema.
7. Риски безопасности.
8. Критерии готовности.
9. Минимальные тестовые сценарии.
10. Документы, которые нужно обновить.
