# Матрица приёмки

Актуально на 2026-07-05.

| Область | Критерий приёмки | Статус |
| --- | --- | --- |
| Обработанный голос | Полная запись с товаром, количеством, ценой, суммой и `confidence >= 0.80` сохраняется как `processed` и входит в отчёт. | Покрыто тестами парсера/отчёта. |
| Разделение парсера | `Сникерс, 3 штуки по 200 рублей. Буханка хлеба, 5 штук по 50 рублей.` создаёт две `sale_items` и сумму `850`. | Покрыто `tests/sale-parser.test.ts` и smoke-проверкой Supabase RPC из журнала изменений. |
| Резервный парсер | Невалидный JSON от LLM или восстанавливаемый сбой парсера создаёт проверяемые позиции, а не теряет продажу. | Покрыто тестами резервного парсера. |
| Проверка голоса | Неполная или низкоуверенная запись сохраняется как `needs_review` и не включает неполные позиции в выручку. | Покрыто тестами статусов/отчёта. |
| Клавиатура Telegram | Сообщение проверки содержит только `✅ Подтвердить` и `❌ Отмена`. | Покрыто регрессией клавиатуры. |
| Данные callback | Новые данные callback имеют формат `confirm:<id>` и `cancel:<id>`, устаревший префикс принимается. | Покрыто тестами разбора callback. |
| Доставка webhook | Продакшен webhook установлен с `allowed_updates: ["message", "callback_query"]`. | Покрыто скриптом/документацией; требует live webhook-info после развертывания. |
| Подтверждение полной продажи | Подтверждение переводит все валидные активные позиции в `processed`, продажу/голосовую запись в `processed`, пересчитывает `total_amount`. | Покрыто сервисной регрессией. |
| Подтверждение смешанной продажи | Подтверждение подтверждает валидные активные позиции, оставляет неполные активные позиции `needs_review`, родительская продажа может остаться `needs_review`. | Покрыто регрессией смешанной корзины. |
| Подтверждение без валидной позиции | Если нет полной позиции, подтверждение возвращает `Не удалось подтвердить: нет ни одной полной позиции.` и не меняет строки. | Покрыто регрессией no-confirmable. |
| Идемпотентность подтверждения | Повторное подтверждение уже обработанной продажи возвращает `✅ Уже подтверждено`. | Покрыто регрессией идемпотентности. |
| Отмена | Отмена продажи на проверке переводит продажу/голосовую запись в `cancelled`, активные позиции в `excluded` + `deleted_at`, сумма становится 0. | Покрыто сервисной регрессией. |
| Идемпотентность отмены | Повторная отмена уже отменённой продажи возвращает успех без изменений. | Покрыто сервисной регрессией. |
| Изоляция | Callback чужого продавца/магазина не меняет запись. | Покрыто регрессионным тестом. |
| Выручка отчёта | Отчёт считает только активные строки `processed` с `deleted_at is null`, валидным количеством/весом и суммой. | Покрыто тестами отчёта. |
| Выручка родительской проверки | Активная позиция `processed` внутри родительской продажи `needs_review` входит в выручку, если родитель не `cancelled`/`failed`. | Покрыто тестами выручки от 2026-07-03. |
| Исключения | `needs_review`, `needs_price`, `failed`, `excluded`, удалённые строки, родительские `cancelled`/`failed` не входят в выручку. | Покрыто тестами отчёта/статусов. |
| Редактирование WebApp | Карточка открывает режим редактирования, сохраняет `product_name`, `quantity`, `unit`, `price`, пересчитывает `total`, ставит позицию `processed`. | Реализовано в `SaleItemCard`, `updateSaleItemAction`, `buildManualSaleItemPatch`. |
| Выручка после редактирования проверки | Валидное редактирование позиции проверки может сделать конкретную строку позиции `processed` и включить её в выручку без скрытого подтверждения всей продажи. | Покрыто тестами save-then-confirm/revenue. |
| Удаление WebApp | Корзина показывает подтверждение `Удалить товар из отчёта?`, выполняет мягкое удаление без физического удаления. | Реализовано и покрыто тестами patch/report. |
| Восстановление | Мягко удалённая позиция может быть восстановлена с предыдущим статусом и пересчётом продажи. | Реализовано; перед релизом нужна ручная smoke-проверка. |
| Сброс дня | Сброс дня мягко удаляет активные позиции одного дня и не удаляет `sales`. | Реализовано; требует продуктовой smoke-проверки перед релизом. |
| Страница проверки | `/review` показывает активные позиции проверки, поддерживает подтверждение, отмену и массовое подтверждение через серверные действия. | Реализовано и покрыто тестами сервиса/статусов. |
| Журнал записей | `/records` показывает журнал голосовых продаж, поиск, фильтр продавца, аудиоссылку при наличии и раскрытие товаров. | Реализовано; покрыто тестами записей. |
| Страница продавцов | `/sellers` показывает активность, количество записей, последнюю активность и выручку за период. | Реализовано; покрыто логикой отчёта/scope. |
| Авторизация магазина | `shop_id` не принимается от клиента; WebApp и callback используют серверный контекст. | Покрыто тестами auth/scope. |
| Привязка владельца | Активный владелец без продавца может создать привязку продавца только в магазине владельца. | Покрыто тестами Telegram principal. |
| Резервный режим | Браузерный fallback требует `ALLOW_WEBAPP_FALLBACK`, `DEFAULT_SHOP_ID`, `DEFAULT_SELLER_ID` и совпадающий магазин продавца. | Покрыто тестами сессии WebApp. |
| Диагностика | `/debug-telegram` скрыт без `DEBUG_TELEGRAM_WEBAPP=true` в production. | Покрыто тестами auth/diagnostics. |
| Ошибки | Ошибки auth/БД отображаются явно и не превращаются в пустой отчёт. | Реализовано в сообщениях API записей; проверять smoke-проверкой. |
| Секреты | Raw initData, токен бота, секрет webhook и service role key не логируются. | Проверяется дисциплиной review/log; требует проверки логов развертывания. |

## Задел smoke-проверки релиза

Перед production-релизом отдельно проверить:

1. Реальное открытие WebApp через Telegram кнопку.
2. Реальный `initDataLength > 0`.
3. Совпадение seller/shop в логах авторизации и отчёта.
4. `npm run telegram:webhook-info` на production-окружении.
5. Сценарий обработанной голосовой записи.
6. Сценарий голосовой записи на проверке.
7. Подтверждение в Telegram.
8. Отмена в Telegram.
9. Подтверждение проверки в WebApp.
10. Редактирование/удаление/восстановление в WebApp.
