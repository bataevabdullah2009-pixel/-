# Обработка голоса

Обработка голоса превращает голосовое сообщение Telegram в структурированные `sale_items`. Основная цель - не потерять продажу и не включить сомнительные позиции в выручку без проверки.

## Конвейер

```text
Голос Telegram
  -> определение продавца
  -> скачивание файла Telegram
  -> подготовка аудио
  -> загрузка аудио по возможности
  -> русская STT
  -> очистка расшифровки
  -> LLM-парсер продажи
  -> детерминированная проверка доказательств
  -> резервное разделение позиций
  -> Supabase RPC save_voice_sale
  -> проверка обратным чтением
  -> ответ Telegram
```

## Определение продавца

1. Бот получает Telegram user id из `ctx.from.id`.
2. `requireSeller()` ищет активного продавца по `telegram_id`.
3. Если продавец неактивен, голосовая запись не сохраняется.
4. Если продавец отсутствует и демо-режим выключен, пользователь получает сообщение о непривязанном Telegram.
5. Если демо-режим включён, продавец может быть создан в магазине по умолчанию.
6. Все последующие записи получают `seller.id` и `seller.shopId` с сервера.

## Обработка аудио

1. Бот скачивает аудио Telegram по ссылке файла.
2. `prepareTelegramVoiceForStt()` пытается конвертировать OGG в MP3 через `ffmpeg-static`.
3. Если ffmpeg недоступен, STT получает исходный OGG.
4. Если конвертация падает, STT получает исходный OGG.
5. Диагностика конвертации логируется без секретов.
6. Загрузка исходного голоса в Supabase Storage выполняется по возможности.
7. Ошибка загрузки не блокирует продажу.
8. Журнал записей использует signed URL, если `audio_path` сохранён.

## STT и очистка

1. STT вызывается с подготовленным аудиофайлом.
2. Сырая расшифровка сохраняется в `voice_records.raw_text`.
3. Сырая расшифровка пишется в audit log по возможности.
4. Cleanup LLM исправляет регистр и пунктуацию.
5. Cleanup не должен добавлять товар, цену или количество.
6. Если запрос очистки падает, используется простая очистка.
7. Пустая STT-расшифровка переводится в резервную проверку.

## Парсер

LLM-парсер должен вернуть JSON по схеме:

```json
{
  "items": [
    {
      "product_name": "string",
      "quantity": 1,
      "unit": "шт",
      "price": 100,
      "total": 100,
      "confidence": 0.95
    }
  ],
  "raw_text": "string",
  "cleaned_text": "string",
  "needs_review": false
}
```

Правила парсера:

1. Не выдумывать товар.
2. Не выдумывать количество.
3. Не выдумывать цену.
4. Не включать количество/цену в `product_name`.
5. Сохранять порядок товаров.
6. Ставить низкую уверенность при неоднозначности.
7. Возвращать `needs_review=true`, если данные неполные.

## Детерминированный слой доказательств

`enforceTranscriptEvidence()` проверяет вывод LLM по расшифровке.

Поддерживается:

1. Цифры.
2. Числа словами.
3. Десятичная запятая.
4. `шт`, `штука`, `штуки`, `штук`.
5. `кг`, `килограмм`, `килограмма`, `килограммов`.
6. `г`, `гр`, `грамм`, `грамма`, `граммов`.
7. `бутылка`, `бутылки`, `бутылок` как `шт`.
8. Цена с `руб`, `рублей`, `₽`.
9. Короткая фраза цены `по 100`.
10. Голое количество перед ценой: `5 по 100`.

Резервный разбор разделяет:

```text
Сникерс, 3 штуки по 200 рублей. Буханка хлеба, 5 штук по 50 рублей.
```

на две позиции:

1. `Сникерс`, `3`, `шт`, `200`, `600`.
2. `Буханка хлеба`, `5`, `шт`, `50`, `250`.

Неполный хвост вроде `Корзина продуктов` сохраняется отдельной позицией на проверку и не блокирует валидные позиции.

## Правило готовности

Позиция получает `processed`, если выполнены условия:

1. `product_name` осмысленный.
2. `quantity > 0` или распознан вес.
3. `price > 0` или есть `total`, из которого можно вывести цену за единицу.
4. `total` валиден.
5. `confidence >= 0.80`.

Позиция получает `needs_review`, если:

1. Нет осмысленного товара.
2. Нет количества или веса.
3. Нет цены и total.
4. Уверенность ниже порога.
5. Резервный парсер не смог подтвердить доказательства.
6. Текст похож на общий контейнер без товара.

## Единицы и суммы

1. `шт` считается как `quantity * price`.
2. `кг` считается как `quantity * price`.
3. `г` считается как `(quantity / 1000) * price`.
4. `бутылки` нормализуются в `шт`.
5. `price` хранит цену за единицу.
6. `total` хранит итог строки позиции.
7. Цена за единицу может быть восстановлена из `total / quantity` при подтверждении.

## Сохранение

`saveVoiceSale()`:

1. Обеспечивает хотя бы одну проверяемую строку позиции.
2. Нормализует каждую позицию.
3. Ищет опциональное совпадение в каталоге товаров.
4. Вычисляет статус позиции.
5. Вычисляет статус продажи.
6. Вычисляет сумму продажи по обработанным позициям.
7. Вызывает RPC `save_voice_sale`.
8. Проверяет возвращённые id.
9. Читает продажу обратно.
10. Читает количество позиций обратно.
11. Сравнивает количество с длиной payload.
12. Пишет audit logs по возможности.

Если обратное чтение не совпало, бот не отправляет успех.

## Ответы Telegram

Уверенная запись:

```text
✅ Запись сохранена: ...
```

Сомнительная запись:

```text
⚠️ Запись сохранена, но нужно подтвердить товары и цены.
Распознано: ...
```

Клавиатура проверки:

1. `✅ Подтвердить`.
2. `❌ Отмена`.

В сообщении проверки нет `Открыть отчёт`.

## Обработка сбоев

Этапы:

1. `seller_resolve`.
2. `telegram_download`.
3. `audio_prepare`.
4. `stt`.
5. `llm`.
6. `supabase_insert`.
7. `telegram_reply`.

Если сбой случился до сохранённой продажи и продавец известен, бот пытается сохранить `voice_records.status = failed` с диагностическим `error_message`.

## Критерии приёмки

1. Полная фраза сохраняется как `processed`.
2. Фраза с несколькими позициями создаёт несколько `sale_items`.
3. Неполный хвост создаёт строку проверки.
4. Невалидный JSON от LLM не теряет продажу.
5. Сбой загрузки аудио не блокирует продажу.
6. Ложный успех Supabase невозможен из-за проверки обратным чтением.
7. Сообщение проверки содержит только две кнопки.
8. `cancelled` и `failed` не попадают в отчёт.
9. Активная обработанная позиция в родительской продаже `needs_review` может входить в выручку.
