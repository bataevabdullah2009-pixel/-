# Отчёт продаж

Экран `Отчёт` находится на `/daily-report` и является главным экраном WebApp. Он показывает состояние магазина за выбранный период и не скрывает ошибки авторизации или Supabase за пустыми метриками.

## Верхний блок

1. Название магазина берётся через `getCurrentShopName()`.
2. Заголовок экрана: `Сводка магазина`.
3. Подзаголовок показывает выбранный период.
4. `RefreshButton` даёт явное обновление.
5. DateFilter поддерживает сегодня, вчера, неделю, месяц и выбранную дату.
6. Для дня доступно действие `Сбросить день`.
7. Сброс дня требует подтверждения.
8. Сброс дня мягко удаляет активные позиции выбранного дня.
9. Сброс дня не удаляет `sales`.
10. Результат мутации показывается через `actionNotice`.

## Метрики

Основные карточки:

1. `Выручка`.
2. `Количество товаров`.
3. `Записей`.
4. `Нужно проверить`.

`Нужно проверить` подсвечивается только когда есть активные позиции проверки. Количество проверок не смешивается с выручкой.

## Данные отчёта

`getReport(filters)`:

1. Вызывает `requireOwner()`.
2. Получает серверный `shopId`.
3. Читает `sales` по `shop_id` и периоду.
4. Читает `sale_items` только по найденным id продаж.
5. Проверяет, что каждая строка продажи принадлежит текущему магазину.
6. Исключает родительские продажи со статусом `cancelled` или `failed`.
7. Передаёт позиции в `buildSalesReport()`.
8. Разделяет активные и удалённые позиции.
9. Возвращает ошибку отдельно от пустого отчёта.

## Правило выручки

В итог входят только позиции, которые одновременно удовлетворяют условиям:

1. Родительская продажа принадлежит текущему магазину.
2. Родительская продажа не `cancelled`.
3. Родительская продажа не `failed`.
4. Статус позиции равен `processed`.
5. `deleted_at is null`.
6. `total` не `null`.
7. Количество или вес валидны.
8. Цена валидна или может быть восстановлена из `total / quantity`.

Не входят:

1. `needs_review`.
2. Устаревший `needs_price`.
3. `failed`.
4. `excluded`.
5. Мягко удалённые строки.
6. Родительская `cancelled`.
7. Родительская `failed`.

Родительская продажа со статусом `needs_review` не блокирует активные позиции `processed`. Это важно для смешанной корзины: валидная позиция уже может входить в выручку, а неполная соседняя позиция остаётся в `Проверке`.

## Блоки экрана

1. `Продажи по дням` - компактная столбчатая диаграмма по активным обработанным позициям.
2. `Топ товаров` - до пяти товаров по выручке.
3. `Последние продажи` - карточки активных обработанных позиций, отсортированные по свежести.
4. `Нужно проверить` - активные позиции проверки, если они есть.
5. `Исключённые товары` - панель деталей для мягко удалённых строк.

## Топ товаров

1. Ключ агрегации использует `product_id`, если он есть.
2. Если `product_id` нет, ключ строится по нормализованному названию.
3. Алиасы товаров нормализуют известные формы вроде `хлеба` -> `хлеб`.
4. Количество суммируется до 3 знаков.
5. Выручка суммируется до 2 знаков.
6. Отображаемое имя показывается в нормальном русском регистре.
7. Удалённые строки и строки проверки не попадают в топ.

## Карточки товаров

1. Обработанная позиция показывает товар, количество, единицу, цену и сумму.
2. Позиция проверки показывает бейдж статуса.
3. Позиция проверки показывает причины: нет цены, нет количества/веса или не удалось выделить товар.
4. Редактирование открывается по карандашу.
5. Удаление открывается по корзине.
6. Состояние формы остаётся в карточке.
7. Ошибка сохранения отображается без очистки пользовательского ввода.
8. Успешное сохранение вызывает `router.refresh()`.

## Редактирование в отчёте

Обработанную позицию можно исправить карандашом. После сохранения:

1. `sale_items.product_name` обновляется.
2. `quantity` обновляется.
3. `unit` обновляется.
4. `price` обновляется.
5. `total` пересчитывается.
6. `confidence` становится `1`.
7. `status` становится `processed`.
8. Родительская продажа пересчитывается.
9. Отчёт/проверка/записи/продавцы проходят revalidate.

Позицию проверки тоже можно исправить карандашом. Если после редактирования позиция стала полной и `processed`, эта конкретная позиция может войти в выручку сразу после refresh. Родительская продажа остаётся `needs_review`, если в ней есть другие активные неполные позиции.

## Удаление и восстановление

Корзина не удаляет строку физически. Она выполняет мягкое удаление:

```text
status = excluded
deleted_at = now()
deleted_reason = excluded_by_owner
deleted_previous_status = previous status
```

Удалённая позиция:

1. Исчезает из активного отчёта.
2. Не входит в выручку.
3. Не возвращается после перезагрузки.
4. Остаётся доступной в панели деталей, если страница показывает удалённые позиции.
5. Может быть восстановлена действием `restoreSaleItemAction`.

Восстановление очищает metadata удаления, возвращает предыдущий статус и пересчитывает родительскую продажу.

## Сброс дня

`resetDayRevenueAction`:

1. Разрешён только для `today`, `yesterday` или `custom`.
2. Проверяет, что диапазон не больше одного дня.
3. Читает продажи текущего магазина за диапазон.
4. Мягко удаляет активные позиции этих продаж.
5. Ставит `deleted_reason = day_reset`.
6. Пересчитывает каждую родительскую продажу.
7. Пишет audit log best-effort.
8. Возвращает читаемое сообщение.

## Пустые состояния и ошибки

1. Нет данных за период - показывается пустое состояние.
2. Ошибка auth - показывается уведомление действия с причиной.
3. Ошибка Supabase - показывается уведомление действия.
4. Состояние ошибки не должно выглядеть как успешный нулевой отчёт.
5. Сбой аудио/storage не ломает сам отчёт.

## Критерии приёмки

1. Отчёт не принимает `shop_id` от клиента.
2. Отчёт показывает только данные текущего магазина, определённого на сервере.
3. Родительская продажа `needs_review` может давать выручку через активную позицию `processed`.
4. Родительские `cancelled` и `failed` не дают выручку.
5. Мягко удалённая позиция не видна как активная.
6. Редактирование меняет строку Supabase и сохраняется после перезагрузки.
7. Удаление сохраняется после перезагрузки.
8. Восстановление возвращает позицию и пересчитывает итоги.
9. Сброс дня не удаляет продажи физически.
10. Ошибка auth/БД не превращается в пустое состояние.
