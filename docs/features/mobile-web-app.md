# Мобильный WebApp

WebApp оптимизирован для Telegram Mini App на телефоне и для явно разрешённого браузерного fallback. Основной пользовательский экран - не лендинг, а рабочая панель отчёта.

## Точки входа

Пользователь открывает WebApp через:

1. Кнопку меню Telegram `Открыть отчёт`.
2. Кнопку reply keyboard `Открыть отчёт`.
3. Inline-кнопку WebApp из `/start`.
4. Прямой браузерный fallback только когда серверное окружение это разрешает.

Голосовое сообщение на проверке не содержит кнопку `Открыть отчёт`; под ним остаются только `✅ Подтвердить` и `❌ Отмена`.

## Bootstrap авторизации

Режим Telegram:

1. Root layout загружает Telegram SDK.
2. `TelegramAuthBootstrap` ждёт `window.Telegram.WebApp`.
3. Клиент проверяет raw `initData`.
4. Клиент проверяет `initDataUnsafe.user.id`.
5. Клиент отправляет raw initData в `/api/auth/telegram`.
6. Сервер проверяет HMAC.
7. Сервер ставит httpOnly cookie.
8. Клиент перезагружается после установки сессии.

Резервный режим:

1. Raw initData отсутствует.
2. Сервер проверяет `ALLOW_WEBAPP_FALLBACK=true`.
3. Сервер читает `DEFAULT_SHOP_ID`.
4. Сервер читает `DEFAULT_SELLER_ID`.
5. Сервер загружает продавца.
6. Сервер проверяет, что продавец активен.
7. Сервер проверяет, что магазин продавца совпадает с fallback-магазином.
8. UI получает режим `fallback`.

## Ошибки авторизации

WebApp явно показывает:

1. Отсутствующий initData.
2. Невалидный/истёкший initData.
3. Продавец не привязан.
4. Продавец неактивен.
5. Магазин не найден.
6. Ошибка конфигурации fallback.
7. Ошибка загрузки Supabase.

Ошибка не заменяется нулевым отчётом.

## Навигация

Нижняя навигация:

1. `Отчёт`.
2. `Проверка`.
3. `Записи`.
4. `Продавцы`.

Навигация должна быть компактной и доступной большим пальцем. Диагностика не входит в обычную навигацию.

## Визуальная система

Текущий стиль:

1. Фон `#0B1020`.
2. Поверхность `#12192B`.
3. Вторичная поверхность `#161F34`.
4. Основной акцент `#5B8CFF`.
5. Warning `#F59E0B` только для проверки.
6. Danger `#EF4444` только для разрушительных действий.

Цель интерфейса - спокойная рабочая SaaS-панель, а не промо-страница.

## Мобильная раскладка отчёта

На мобильном экране:

1. Метрики раскладываются компактно.
2. DateFilter остаётся в горизонтальной строке.
3. Диаграмма не растягивает один столбец на весь экран.
4. Топ товаров остаётся читаемым.
5. Последние продажи показываются компактными карточками.
6. Блок проверки не смешивается с выручкой.
7. Удалённые позиции прячутся в details panel.

## Мобильная раскладка проверки

Вкладка `Проверка`:

1. Показывает отдельные карточки для активных позиций проверки.
2. Показывает причины проверки.
3. Содержит `Подтвердить`.
4. Содержит `Отмена`.
5. Содержит `Подтвердить всё`, если есть sale ids.
6. Использует серверные действия.
7. Показывает результат мутации через notice.
8. Не требует ручной перезагрузки после успешной операции.

Смешанная продажа подтверждает валидные позиции и оставляет неполные на проверке.

## UX карточки позиции продажи

1. Области касания для кнопок действий должны быть около 44 px.
2. Edit открывается только по карандашу.
3. Delete открывается только по корзине.
4. В обычном режиме нет больших постоянных кнопок действий под карточкой.
5. Форма раскрывается внутри карточки.
6. Ошибка сохранения не очищает ввод.
7. Pending-состояние блокирует повторную отправку.
8. Delete требует локальное подтверждение.

## Записи и продавцы

Записи:

1. Поддерживает период.
2. Поддерживает фильтр продавца.
3. Поддерживает поиск.
4. Показывает ссылку на аудио при наличии.
5. Раскрывает товары.
6. Не показывает confirm/cancel.

Продавцы:

1. Показывает состояние активности/неактивности.
2. Показывает последнюю активность.
3. Показывает количество записей.
4. Показывает выручку.
5. Использует тот же DateFilter.

## Диагностика

`/debug-telegram`:

1. Доступен в development.
2. Доступен в production только при `DEBUG_TELEGRAM_WEBAPP=true`.
3. Не связан из обычной навигации.
4. Не показывает raw initData.
5. Не показывает секреты.

Логи авторизации могут содержать только безопасные metadata: длину initData, Telegram user id, seller id, shop id, режим, причину ошибки.

Production smoke может автоматически проверить корректно подписанную сессию, cookie, server-rendered `/daily-report`, `/review`, `/records`, `/sellers`, Next assets и Telegram SDK. Наличие `window.Telegram.WebApp` и передачу реального initData окончательно подтверждает только ручной запуск через кнопку Telegram на поддерживаемом клиенте.

## Критерии приёмки

1. WebApp открывается внутри Telegram.
2. Валидный initData устанавливает сессию.
3. Прямой браузер показывает fallback только когда он включён.
4. Отсутствующая сессия показывает понятное сообщение доступа.
5. Навигация имеет ровно четыре основных раздела.
6. Карточки отчёта помещаются в мобильный viewport.
7. Действия проверки удобны на мобильном.
8. Редактирование/удаление не сдвигают раскладку непредсказуемо.
9. Ошибки auth/БД остаются видимыми.
10. Диагностика остаётся закрытой в production.
