# Правила проекта

Эти правила обязательны для человека и агента. При конфликте удобства с безопасностью или границами MVP действует более строгое правило.

| Документ | Область |
| --- | --- |
| [Границы MVP](./mvp-boundaries.md) | Допустимый продуктовый scope и обязательный backlog для новых идей. |
| [AI-парсер](./ai-parser-rules.md) | Запрет догадок, confidence, нормализация и тесты. |
| [База данных](./database-rules.md) | Миграции, вычисления, soft delete, аудит и статусы. |
| [Безопасность](./security-rules.md) | Секреты, service role, RLS и production-доступ. |
| [Развёртывание](./deployment-rules.md) | Порядок migration/deploy, webhook и внешние проверки. |
| [Документация](./documentation-rules.md) | Источники правды, ссылки, язык и отсутствие дублей. |
| [Инженерия](./engineering.md) | Процесс изменения и команды качества. |
| [Отчёты](./reports.md) | Допустимые позиции, группировка и итоги. |
| [Git и GitHub](./git-and-github.md) | Ветки, коммиты, PR, CI и откат. |

Главное правило scope: если новая задача не входит в MVP, она должна идти в `docs/plans/backlog`, а не сразу в код.
