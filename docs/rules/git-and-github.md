# Работа с Git и GitHub

## Назначение

Git хранит локальную историю проекта и позволяет безопасно разделять изменения по веткам. GitHub хранит удалённую копию репозитория, ветки, Pull Request, обсуждения и результаты автоматических проверок.

Текущий локальный репозиторий использует удалённый адрес `origin` на GitHub. Рабочая ветка `codex/stabilize-voice-parsing` отслеживает одноимённую ветку `origin/codex/stabilize-voice-parsing`.

## Основные понятия

| Понятие | Значение в проекте |
| --- | --- |
| Рабочее дерево | Текущие изменённые и новые файлы на компьютере. |
| Индекс | Точный набор изменений, подготовленный к следующему коммиту. |
| Коммит | Проверенная логическая порция изменений с описанием. |
| Ветка | Изолированная линия работы над одной задачей. |
| `origin` | Удалённый GitHub-репозиторий. |
| Pull Request | Запрос на проверку и объединение ветки. |
| Основная ветка | Стабильная ветка, в которую попадают проверенные изменения. Её точное имя определяется настройками GitHub. |

## Рекомендуемый процесс задачи

1. Получить актуальное состояние удалённого репозитория:

   ```bash
   git fetch origin
   git status --short --branch
   ```

2. Создать отдельную ветку от актуальной основной ветки:

   ```bash
   git switch -c feature/kratkoe-nazvanie
   ```

3. Внести изменения, не затрагивая несвязанные пользовательские файлы.
4. Проверить рабочее дерево и содержимое изменений:

   ```bash
   git status --short
   git diff
   ```

5. Выполнить проверки проекта:

   ```bash
   npm run lint
   npm run test
   npm run build
   ```

6. Добавить только относящиеся к задаче файлы и проверить индекс:

   ```bash
   git add <файлы-задачи>
   git diff --cached
   ```

7. Создать один или несколько логических коммитов:

   ```bash
   git commit -m "Обновить русскую документацию проекта"
   ```

8. Отправить ветку в GitHub:

   ```bash
   git push -u origin feature/kratkoe-nazvanie
   ```

9. Создать Pull Request, описать проблему, решение, проверки, миграции и непроверенные внешние шаги.
10. После замечаний добавить новые коммиты в ту же ветку. Не переписывать опубликованную историю без необходимости.
11. После одобрения и успешных проверок объединить Pull Request через GitHub.

## Требования к веткам и коммитам

- Одна ветка решает одну связанную задачу.
- Рекомендуемые префиксы: `feature/`, `fix/`, `docs/`, `refactor/`, `test/`.
- Имя ветки пишется латиницей, в нижнем регистре, слова разделяются дефисами.
- Коммит содержит законченное и проверяемое изменение.
- Сообщение коммита кратко отвечает на вопрос «что сделано».
- Не смешивать форматирование всего проекта, новую функцию и исправление несвязанной ошибки в одном коммите.
- Не добавлять сгенерированные каталоги `node_modules`, `.next`, `coverage` и локальные журналы.

## Содержимое Pull Request

Описание Pull Request должно включать:

- зачем требуется изменение;
- что изменено;
- какие файлы, API, таблицы и пользовательские сценарии затронуты;
- результаты `lint`, `test` и `build`;
- новую миграцию и порядок её применения, если менялась схема;
- снимки экрана или результат браузерной проверки для интерфейса;
- внешние шаги Vercel, Supabase и Telegram, которые выполнены или остались непроверенными;
- известные риски и способ отката.

Pull Request не должен утверждать, что рабочая интеграция проверена, если выполнялись только локальные тесты.

## Секреты и запрещённые файлы

Никогда не добавлять в Git:

- `.env.local` и любые файлы с рабочими секретами;
- токены Telegram, ключи Supabase, STT или LLM;
- выгрузки БД с реальными данными;
- приватные аудиозаписи и расшифровки пользователей;
- временные файлы сборки и зависимости.

Перед коммитом проверить `git diff --cached`. Если секрет попал в опубликованную историю, простого удаления новым коммитом недостаточно: ключ нужно немедленно отозвать, заменить и очистить историю по отдельной процедуре инцидента.

## Миграции и порядок объединения

- Каждое изменение схемы оформляется новым SQL-файлом в `supabase/migrations`.
- Миграция входит в тот же Pull Request, что и зависящий от неё код и документация.
- Совместимость и порядок применения описываются в Pull Request.
- В рабочем окружении миграция применяется до кода, который требует новые столбцы.
- Уже применённую миграцию нельзя тихо переписывать; исправление оформляется следующей миграцией.

## Конфликты и обновление ветки

Перед объединением ветку нужно обновить относительно основной ветки. Конфликт разрешается осознанно по содержанию обоих вариантов, после чего повторяются все проверки.

```bash
git fetch origin
git merge origin/<основная-ветка>
```

Для общей ветки безопаснее обычный merge. Rebase и принудительная отправка допустимы только при явном соглашении команды. Запрещено использовать `git reset --hard` или `git push --force` для удаления чужих изменений.

## Откат

- Ошибку в общей истории откатывать новым `git revert <commit>`, а не переписыванием истории.
- Откат кода, который зависит от миграции, оценивается отдельно: обратное изменение БД может быть небезопасным.
- Перед откатом рабочего окружения проверить совместимость приложения, схемы и данных.
- Причина и результат отката фиксируются в Pull Request или связанном Issue.

## GitHub Actions и защита ветки

В текущем репозитории каталог `.github/workflows` отсутствует. Поэтому GitHub автоматически не запускает lint, тесты и сборку. До настройки CI проверки выполняются локально и записываются в Pull Request.

Рекомендуемая обязательная конфигурация GitHub:

- workflow на Pull Request и отправку в основную ветку;
- шаги установки зависимостей через `npm ci`, затем `npm run lint`, `npm run test`, `npm run build`;
- запрет объединения при неуспешных проверках;
- обязательное рассмотрение Pull Request хотя бы одним участником;
- запрет прямой отправки в основную ветку;
- проверка зависимостей и сканирование случайно добавленных секретов;
- отдельная проверка миграций на временной БД.

Добавление CI находится в дорожной карте и не должно считаться реализованным до появления workflow и успешного запуска в GitHub.

## Связь GitHub с Vercel

Рекомендуемый процесс: Pull Request создаёт предварительное развёртывание Vercel, а объединение в основную ветку — рабочее развёртывание. Это зависит от настроек проекта Vercel и сейчас не подтверждается файлами репозитория.

Даже при автоматическом развёртывании миграции Supabase и Telegram webhook требуют отдельного контролируемого шага. Нельзя рассчитывать, что обычный `git push` автоматически применит миграцию или переустановит webhook.

## Правила для ИИ-агента

- Не создавать коммит, не отправлять ветку и не открывать Pull Request без прямого запроса пользователя.
- Перед любым коммитом показать точный состав изменений через `git status` и `git diff --cached`.
- Не включать несвязанные пользовательские изменения.
- Не использовать разрушительные команды для очистки рабочего дерева.
- Не изменять чужой Pull Request или Issue без указания цели.
- При публикации честно перечислить выполненные и непроверенные шаги.
