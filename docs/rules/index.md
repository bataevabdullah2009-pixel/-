# Правила

Короткие правила разработки проекта. Полные системные правила продукта, статусов, проверки и выручки находятся в [../specs/global.md](../specs/global.md).

## Голосовой конвейер

- Не переписывать STT/парсер/webhook без прямой задачи.
- Не терять голосовую запись при восстанавливаемом сбое парсера.
- Резервный парсер обязан разделять склеенную multi-item расшифровку на отдельные строки.
- Уверенная запись остаётся `processed`.
- Сомнительная запись остаётся `needs_review` до решения проверки в Telegram или WebApp.

## Telegram

- Сообщение проверки содержит только `✅ Подтвердить` и `❌ Отмена`.
- Не добавлять `Открыть отчёт` в сообщение проверки.
- Данные callback: `confirm:<sale_id>` и `cancel:<sale_id>`.
- Callback должен быть идемпотентным.
- Подтверждение подтверждает валидные активные позиции и не блокирует смешанную корзину из-за неполных позиций.
- Webhook должен принимать `message` и `callback_query`.

## WebApp

- Навигация: `Отчёт`, `Проверка`, `Записи`, `Продавцы`.
- `/review` показывает только активные `needs_review` и подтверждает/отменяет через серверные действия.
- `needs_review` показывать как состояние проверки для неполных позиций; соседние обработанные позиции могут быть в выручке.
- Карточка товара компактная: режим просмотра, `✏️`, `🗑`.

## Данные

- Выручка только из активной позиции `processed`, если родительская продажа не `cancelled` и не `failed`.
- Выручка требует валидный `total`; цена за единицу может быть сохранена или вычислена из total.
- Обработанная позиция внутри родительской продажи `needs_review` входит в выручку; неполные позиции той же записи остаются на проверке.
- Мягкое удаление только через `deleted_at`.
- `shop_id` не брать от клиента как источник прав.

## Документы и проверки

- После каждого изменения кода агент обязан обновлять документацию, журнал изменений, активные планы и технические спецификации так, чтобы они соответствовали фактическому состоянию проекта. Запрещено оставлять документацию, противоречащую текущему коду.
- После БД обновлять миграции и спек БД.
- После UI обновлять продуктовые спеки.
- После сценария Telegram обновлять спеки Telegram.
- После работы запускать lint/test/build.
