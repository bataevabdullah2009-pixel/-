# Правила проекта

## Основные правила

- Проект должен оставаться простым.
- Любая новая функция должна соответствовать цели проекта.
- Нельзя добавлять склад, кассу, цены как кассовый модуль и CRM без отдельного решения.
- Таблица `products` нужна только для цены по умолчанию, если продавец не назвал цену.
- Нельзя превращать `products` в полноценный склад, каталог или систему остатков.

## Данные

- Все записи должны иметь `created_at`.
- Все голосовые записи должны иметь `seller_id`, если продавец найден или создан.
- Все продажи должны иметь `seller_id`, если продавец найден или создан.
- Все ошибки должны логироваться в `audit_logs`.
- Нельзя удалять записи физически, только менять статус.
- Все спорные записи должны быть доступны для ручного исправления.

## ИИ

- ИИ не должен выдумывать цены.
- ИИ не должен выдумывать товары.
- ИИ не должен выдумывать категории.
- ИИ не должен делать сложную аналитику.
- Если ИИ не уверен в тексте, статус должен быть `needs_review`.
- Если цена неизвестна, позиция должна получить статус `needs_price`.
- Если количество не названо, можно использовать `quantity = 1`, но confidence должен быть ниже.

## Отчёты

- Отчёт не должен показывать лишние технические данные.
- В дневном отчёте по умолчанию не показывать продавца и время.
- Основная таблица отчёта должна содержать только товар, количество и выручку.
- Все неизвестные цены должны попадать в “Нужно проверить”.
- Система должна считать выручку только по подтверждённым данным.
- В итоговую сумму нельзя включать позиции без цены.
- Одинаковые товары в отчёте должны группироваться.

## Безопасность

- Нельзя хранить токены в коде.
- Все переменные окружения должны быть в `.env.example`.
- Локальные секреты должны быть в `.env.local`.
- `SUPABASE_SERVICE_ROLE_KEY` нельзя использовать в клиентских компонентах.
- В Supabase таблицы в `public` должны иметь RLS.

## Документация и сопровождение

- Все изменения фиксировать в `CHANGELOG.md`.
- Все новые функции описывать в `docs/specs.md`.
- Все готовые задачи отмечать в `docs/workplan.md`.
- Если меняется архитектура, обновлять `docs/architecture.md`.
- Если меняется структура папок, обновлять README и architecture.
