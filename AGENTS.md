# AGENTS.md

Строгие правила для любого ИИ-агента, который меняет проект `voice-sales-log`.

## Язык и названия документов

- Вся проектная документация пишется на русском языке: корневое описание, спецификации, планы, правила, пользовательские истории, архитектура, дорожная карта и журнал изменений.
- Английский допустим только для имён файлов и каталогов, программных идентификаторов, команд, названий технологий и точных значений API/БД.
- Не создавать новые файлы с именем `README.md`. Для документа использовать содержательное имя по назначению: например, `architecture.md`, `roadmap.md`, `plan.md` или `verification.md`.
- Существующий корневой `README.md` можно обновлять, но нельзя создавать дополнительные `README.md` в новых каталогах.
- Не создавать дублирующие документы. Перед добавлением файла проверить карту документации и обновить существующий профильный документ, если он уже есть.

## Обязательное чтение перед изменениями

Прочитать полностью и в указанном порядке:

1. `README.md`;
2. `docs/overview/README.md`;
3. `docs/specs/global.md`;
4. `docs/specs/technical/README.md`;
5. `docs/rules/README.md`;
6. каждый файл `docs/plans/active/*/plan.md`;
7. `codex/skills/voice-sales-log/SKILL.md`.

Каталог `codex-skills/voice-sales-log` оставлен только для обратной совместимости и должен направлять к основному навыку в `codex/skills/voice-sales-log`.

## Где искать требования

- Обзор документации: `docs/overview/README.md`.
- Продукт и пользовательский сценарий: `docs/specs/global.md`, `docs/specs/product/`.
- Технические контракты: `docs/specs/technical/`.
- Модель данных, статусы и аудит: `docs/specs/data/`.
- Архитектура: `docs/architecture/architecture.md`.
- Обязательные ограничения: `docs/rules/README.md` и дочерние файлы.
- Работа с ветками, коммитами и Pull Request: `docs/rules/git-and-github.md`.
- Активная работа: `docs/plans/active/*`.
- Завершённые этапы: `docs/plans/completed/*`.
- Будущие задачи: `docs/plans/backlog/*`.
- Дорожная карта: `docs/roadmap/roadmap.md`.
- История изменений: `CHANGELOG.md`.

## Непереговорные запреты

- Не превращать проект в CRM, склад, кассу, систему остатков, фискализации или онлайн-оплаты.
- Не менять Telegram webhook без подтверждённой причины и проверки контракта секретного заголовка.
- Не менять парсер без регрессионных тестов.
- Не менять схему БД без новой миграции.
- Не коммитить, не читать без необходимости и не выводить `.env.local`, токены, ключи API и другие секреты.
- Не использовать `SUPABASE_SERVICE_ROLE_KEY` в клиентском компоненте или браузерном пакете.
- Не запускать polling на Vercel или одновременно с рабочим webhook.
- Не удалять физически голосовую запись, аудио или продажу при исключении товара; использовать документированное мягкое удаление.
- Не выдавать демонстрационные анонимные политики за производственную авторизацию.
- Не отмечать проверку Vercel, Supabase или Telegram выполненной без фактического доступа и проверки.
- Не описывать планируемую возможность как уже реализованную.

## Инварианты реализации

- Webhook и локальный polling используют общий `processTelegramUpdate`.
- Парсер не придумывает значения; итог позиции пересчитывается приложением.
- Отчёт строится из активных допустимых `sale_items`.
- Ручные изменения пересчитывают родительскую продажу и создают запись аудита.
- Миграция применяется до развёртывания кода, который зависит от новых столбцов.
- Исходные `raw_text`, `parser_json` и голосовая запись не переписываются ручной корректировкой товара.

## Правила изменения документации

- Документация должна описывать фактическое состояние кода, миграций и внешних настроек.
- Спецификация описывает действующий контракт, план — ещё не завершённую работу, дорожная карта — порядок и критерии этапов.
- При изменении ссылки обновить все входящие ссылки и проверить отсутствие ссылок на удалённый путь.
- При изменении архитектуры обновить `docs/architecture/architecture.md`.
- При изменении приоритетов или готовности обновить `docs/roadmap/roadmap.md`.
- Не смешивать русский и английский в заголовках и обычном тексте.

## Завершение работы

1. Обновить затронутые спецификации, функции, истории и правила.
2. Обновить `CHANGELOG.md` на русском языке.
3. Обновить контрольный список и заметки активного плана.
4. Если задача полностью завершена, оформить результат в `docs/plans/completed/` и убрать задачу из активных.
5. Проверить внутренние ссылки и отсутствие случайно созданных `README.md`.
6. Запустить `npm run lint`.
7. Запустить `npm run test`.
8. Запустить `npm run build`.
9. Для изменений интерфейса выполнить проверку в браузере; для интеграций явно указать непроверенные внешние шаги.
