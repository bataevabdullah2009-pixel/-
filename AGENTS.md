# AGENTS.md

Главная инструкция для ИИ-агента, который работает с `voice-sales-log`.

## Обязательное чтение перед любыми изменениями

## Reading order for AI agents

Always read in this order:

1. README.md
2. docs/INDEX.md
3. docs/specs/global.md
4. docs/specs/technical/README.md
5. docs/specs/product/production-readiness.md
6. docs/rules/README.md
7. docs/plans/active/README.md
8. codex/skills/voice-sales-log/SKILL.md

Do not start implementation before reading these files.

После этого прочитать профильную спецификацию затрагиваемой области и проверить Git-состояние.

## Источники правды

- `README.md` — общий обзор и запуск.
- `docs/INDEX.md` — карта документации.
- `docs/specs/global.md` — главная спецификация.
- `docs/specs/product/*` — продуктовые границы, роли и production-readiness.
- `docs/specs/technical/*` — технические требования.
- `docs/features/*` — описание функций.
- `docs/rules/*` — обязательные правила.
- `docs/plans/*` — активная работа, завершённые результаты и backlog.
- `docs/architecture/*` — архитектура и технический долг.
- `codex/skills/voice-sales-log/SKILL.md` — инструкции Codex.

Спецификация описывает контракт, план — незавершённую работу, roadmap — последовательность этапов. При расхождении с кодом агент обязан зафиксировать расхождение, а не объявлять планируемое реализованным.

## Организация планов и документов

- В `docs/plans/active` одновременно находится не более одного предметного плана.
- Следующий этап хранится в backlog; выполненный этап переносится в completed.
- `docs/INDEX.md` — единственная глобальная карта; тематические `README.md` индексируют только свой раздел.
- Не создавать дублирующие документы. При переименовании обновить все ссылки и удалить устаревший файл.
- Проектная документация пишется на русском; английский допустим для имён, команд, технологий и API/БД.

## Запреты

- Не добавлять CRM.
- Не добавлять склад, остатки или движения товара.
- Не добавлять кассу, фискализацию или онлайн-оплату.
- Не менять Telegram webhook без подтверждённой причины и проверки секретного заголовка.
- Не менять LLM-парсер без регрессионных тестов.
- Не менять Supabase schema без новой migration.
- Не коммитить и не выводить `.env.local` или другие секреты.
- Не использовать `SUPABASE_SERVICE_ROLE_KEY` на клиенте.
- Не запускать polling на Vercel или одновременно с рабочим webhook.
- Не выдавать демонстрационные RLS-политики за production-авторизацию.
- Не отмечать внешнюю проверку выполненной без фактического доступа.

## Инварианты

- Webhook и polling используют общий `processTelegramUpdate`.
- LLM не придумывает значения, итог позиции считает приложение.
- Отчёт строится из активных допустимых `sale_items`.
- Ручная правка не переписывает `raw_text`, `parser_json` или исходное аудио.
- Мягкое удаление обратимо, а мутации пересчитывают продажу и создают аудит.
- Миграция применяется до кода, который от неё зависит.

## После изменений

1. Обновить `CHANGELOG.md`.
2. Обновить specs, если изменилась логика или контракт.
3. Обновить plans: выполненное перенести в completed, следующий этап активировать только при необходимости.
4. Проверить внутренние ссылки, пустые папки и отсутствие случайных секретов.
5. Запустить `npm run lint`.
6. Запустить `npm run test`.
7. Запустить `npm run build`.
8. Для UI выполнить браузерную проверку; для внешних интеграций перечислить непроверенные шаги.

## Documentation sync is mandatory.

После любого изменения кода агент обязан:

1. Проверить, какие docs затронуты.
2. Обновить specs, если изменилась логика.
3. Обновить database docs, если изменилась schema.
4. Обновить feature docs, если изменилась фича.
5. Обновить plans, если задача завершена.
6. Обновить `CHANGELOG.md`.
7. Запустить `npm run test`.
8. Запустить `npm run build`.

Если документация не обновлена, задача считается незавершённой.
